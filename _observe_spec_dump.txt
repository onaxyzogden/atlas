1. Context & architectural decisions



1.1 What Observe is

Observe is the read-only intelligence layer of OLOS. It surfaces the synthesised state of the land as produced by verified Act tasks and divergence evidence. The steward never ‘does’ Observe — they read it. Its richness grows as Act tasks complete across cycles.



Core principle

Observe records reality. It is not a plan, a to-do list, or a progress tracker. It is an evidence-based picture of the land as it actually is right now — built from proof collected in the field, not from what was intended.



Two categories of data populate Observe:



Data source

How it enters Observe

Verified Act tasks

Immediately on verification. Populates the relevant domain data point with proof items, measurements, logged results, GPS traces, and notes from the task’s proof record.

Divergence evidence

Immediately on divergence capture submission, unconditionally. Routes to the relevant domain as an observation record regardless of whether the associated Plan objective has been revised.



1.2 Three surfaces

Observe is organised as three layered surfaces with a clear hierarchy:



Surface

Role

1. Unified Land State

A single dashboard showing the whole project’s current state, data gaps, and freshness across all domains. Default entry point. Answers: “What is the land telling me right now?”

2. Domain Detail Views

Per-domain dashboards showing all observations, measurements, trends, and evidence for one domain. Reached by drilling down from the Unified view. Answers: “What do I know about water on this site?”

3. Temporal Layer

A time-aware view of how any domain’s data has changed across Act cycles. Accessible from any Domain Detail View. Answers: “Is the land responding to our interventions?”



1.3 Two audiences

Observe serves two distinct audiences with different needs:



Audience

Primary need

Access mode

Project steward

Operational intelligence: what has been observed, what is missing, what has changed, and when to revisit Plan

Full Observe access within OLOS

External stakeholder (ecologist, funder, certifier, landowner)

Outcome evidence: ecological trajectory, intervention proof, condition baseline and change over time

Presentation mode — a clean, shareable view of selected Observe data without internal planning state



Audience separation principle

External stakeholders should never see incomplete objectives, locked tiers, divergence flags, or internal planning state. Presentation mode surfaces only verified, timestamped evidence. The same underlying data powers both views — the difference is entirely in what is displayed.

2.3 Data freshness model

Every data point carries a timestamp. The Unified Land State and Domain Detail Views communicate data freshness explicitly so the steward knows how current their picture is.



Freshness state

Definition

UI treatment

Current

Data point observed within the current or most recent Act cycle

No indicator needed — the default assumption

Ageing

Data point from a previous cycle but within a domain-defined freshness threshold

Subtle amber dot on the domain card. Threshold is domain-specific (e.g. soil pH: 12 months; species record: 6 months; photo: no expiry).

Stale

Data point older than the domain freshness threshold

Amber ‘Needs refresh’ badge on the domain card. Surfaces as a suggested Act task in Plan.

Missing

No data point exists for this domain or sub-domain

Grey ‘Not yet observed’ placeholder. Surfaces as a required Act task in Plan.



Freshness thresholds are domain-specific

A photo of a completed swale does not expire. A soil moisture reading from 18 months ago is not reliable for current decisions. Freshness thresholds must be configurable per domain and data type, with sensible defaults. Default thresholds: measurements (6 months), species records (12 months), photos of permanent infrastructure (no expiry), photos of living systems (12 months), logged test results (12 months).




3. Surface 1 — Unified Land State



3.1 Purpose

The Unified Land State is the default entry point for Observe. It gives the steward a single-screen answer to: “What is the land telling me right now?” It must be readable in 30 seconds and actionable in one additional tap.



3.2 Layout

Zone

Content

Top bar

Stage pills (Plan / Act / Observe), project name, cycle indicator (e.g. ‘Cycle 3’), and a ‘Share’ control entering Presentation mode

Land health summary

A compact header row showing: total data points recorded, domains with current data, domains with ageing/stale data, and domains with no data. Not a score — a factual count.

Domain cards grid

One card per domain. 2-column grid on tablet/desktop, single column on mobile. Each card summarises the domain’s current observation state.

Plan revision banner

Shown only when one or more Observe data points have changed enough to warrant Plan review. See section 3.5.

Divergence feed

A compact list of unresolved divergence records from Act, shown below the domain grid. Each item links to the relevant Act task and Plan objective.



3.3 Domain cards

Each domain card on the Unified Land State shows:

Domain name and icon

Freshness indicator: Current (no badge), Ageing (amber dot), Stale (amber ‘Needs refresh’ badge), or Missing (grey ‘Not yet observed’)

Data point count and most recent observation date

One-line summary of the most significant recent observation (e.g. ‘Soil pH 6.4 at Zone 2 — 3 weeks ago’)

A thumbnail of the most recent photo if one exists



Tapping a domain card opens the Domain Detail View for that domain.



No scores or health ratings

The Unified Land State must not display health scores, traffic-light ratings, or aggregated wellness indices for domains. These are seductive but ecologically misleading — a ‘red’ soil domain might reflect an accurate observation of challenging conditions, not a failure. Observe shows evidence; it does not evaluate it. Evaluation belongs in Plan.



3.4 Land health summary

The summary row at the top of the Unified Land State shows four factual counts only:

Total verified data points recorded across all domains

Domains with current data (all data points within freshness threshold)

Domains with ageing or stale data (at least one data point past threshold)

Domains with no data (not yet observed)



These counts link directly to filtered views of the domain grid. Tapping ‘Not yet observed’ filters the grid to show only missing domains. No charts, gauges, or radar diagrams — counts only.



3.5 Plan revision banner

When Observe data crosses a threshold that warrants revisiting a Plan decision, a banner appears at the top of the Unified Land State. This is the primary mechanism by which Observe closes the spiral back into Plan.



Revision triggers:

A divergence record has been open for more than 14 days without a Plan revision flag being resolved

A measurement data point has changed by more than a configurable threshold since the last Plan cycle (e.g. soil pH dropped by more than 0.5 units)

A domain has passed from Ageing to Stale since the last Plan cycle

A new species record has been added that affects a Plan decision (e.g. a protected species found in a proposed planting zone)



The banner shows:

How many Plan objectives are affected

The specific domains and data points that triggered the review

A ‘Review in Plan’ CTA that opens the Plan Command Center at the relevant tier and objective



Trigger priority hierarchy

Multiple triggers may be active simultaneously. Without prioritisation the banner becomes noise. Triggers are ranked into three priority levels:



Priority

Triggers

Banner behaviour

Critical — act immediately

Protected or significant species found in a planned intervention area; divergence record open more than 14 days; safety-related divergence of any age

Banner rendered in amber at full width above the domain grid. Cannot be dismissed without resolving or explicitly deferring the trigger in Plan.

High — review before next Act cycle

Measurement changed beyond threshold since last Plan cycle; new discovery divergence type unresolved

Banner rendered as a standard blue callout. Dismissible for the current session with a ‘Remind me next session’ option.

Informational — review at next Plan cycle

Domain passed from Ageing to Stale; minor measurement drift within acceptable range

Shown as a count in the land health summary row only, not as a full banner. Available in a ‘Pending reviews’ drawer.



When multiple critical triggers are active, they are listed together in a single banner — not as separate banners stacked vertically. The banner shows a count (‘ 3 items need immediate attention’) with an expandable list.



Observe does not make Plan decisions

The Plan revision banner is a signal, not a directive. It surfaces changed evidence and links to Plan — it never recommends a specific design change or automatically updates a Plan objective. The steward decides what to do with the evidence.




4. Surface 2 — Domain Detail Views



4.1 Purpose

Domain Detail Views show everything Observe knows about one domain. They are the primary surface for stewards who are deep in a specific area of land management — reviewing all hydrology evidence before revising the water strategy, or checking all soil data before planning an amendment programme.



4.2 Layout

Zone

Content

Header

Domain name, data point count, freshness state, and a ‘View timeline’ control entering the Temporal Layer for this domain

Map view (prominent)

Full-width map showing all GPS-linked data points for this domain as pins and traces. Filterable by data type and date range.

Data point list

Chronological list of all data points for this domain, most recent first. Each item shows type, value/thumbnail, location, observed date, and source task.

Sub-domain grouping

Data points grouped by sub-domain where applicable (e.g. Hydrology grouped into: Surface Water, Infiltration, Water Sources, Flood Risk). Collapsible.

Divergence records

Any divergence evidence for this domain shown in a distinct section with amber styling. Never mixed with verified task data.



4.3 Data point detail

Tapping any data point in the list expands it to show:

Full proof record: all photos, GPS geometry, measurements, and notes from the source Act task

Source task title and parent Plan objective

Observed at and recorded at timestamps

Cycle in which this data point was produced

Whether this data point is current or superseded by a later observation

A ‘View on map’ control that centres the map on this data point’s location



Superseded data points

When a verified Act task produces a data point at the same domain and location as an existing record, the earlier record is marked is_superseded = true. The UI must make supersession transparent to the steward:

Superseded data points are shown in the data point list with a ‘Superseded’ badge and muted styling. They remain fully accessible — they are never hidden.

The superseding record shows a ‘Replaces an earlier observation’ link that navigates directly to the superseded record.

The superseded record shows a ‘Superseded by’ link pointing to the newer record with its date.

In the Temporal Layer, superseded data points appear as hollow markers on the chart — visually distinct from current data points but present in the timeline so the full history is readable.

Supersession is automatic and system-determined based on domain, location proximity (within a configurable radius, default 10m), and data type match. The steward cannot manually mark a record as superseded.

If a steward disputes an automatic supersession — e.g. two legitimate observations at the same location representing genuinely different conditions — they can mark the newer record as ‘Not a replacement’, which restores the prior record to current status and treats both as independent observations.



4.4 Domain-specific data displays

Some domains have structured data types that warrant purpose-built display components beyond a generic list:



Domain

Specialised display

Hydrology & Water

Runoff path overlays on map, infiltration rate comparison across test locations, water source inventory

Soil & Subsurface

Soil type map by zone, pH and organic matter trend charts across test points, compaction flag heatmap

Ecology & Biodiversity

Species list with observation frequency and last-seen date, vegetation community map, invasive species distribution

Climate & Sectors

Sector compass with observed force intensities, frost pocket map, microclimate zone overlay

Topography & Landform

Contour and slope map built from GPS traces, erosion risk zones, high/low point inventory

Built Infrastructure

Asset register with condition ratings, photo timeline per asset, maintenance history



Domain-specific display components are progressive enhancements. The generic data point list is the baseline. Specialised components layer on top as the domain accumulates sufficient data to make them meaningful.




5. Surface 3 — Temporal Layer



5.1 Purpose

The Temporal Layer answers the most important question in regenerative land management: is the land responding? It shows how any domain’s observations have changed across Act cycles over time. This is the evidence base for adaptive management and for demonstrating ecological outcomes to external stakeholders.



Why this is the most ecologically significant surface in OLOS

A single soil pH measurement is data. Soil pH measured at the same location across five consecutive cycles, showing a trajectory from 5.2 to 6.4 after targeted soil amendments, is evidence of regeneration. The temporal layer transforms OLOS from a project management tool into an ecological monitoring system. It is the surface that makes the case — to funders, certifiers, and stewards alike — that the work is working.



5.2 Entry points

From any Domain Detail View via the ‘View timeline’ control

From a specific data point via a ‘Show history at this location’ control

From the Presentation mode’s trajectory section (see section 6)



5.3 Layout

Zone

Content

Timeline header

Domain name, selected data type, date range selector (by cycle or by calendar), and a location filter

Primary chart

The main temporal display. Chart type adapts to data type (see 5.4). X-axis is always time; Y-axis is the data value.

Cycle markers

Vertical lines on the chart marking the start of each Plan–Act–Observe cycle. Plan decisions and major interventions annotated on the relevant cycle marker.

Photo timeline

A horizontal strip of thumbnail photos below the chart, ordered chronologically, filtered to the same domain and location. Shows visual change over time independently of the chart.

Comparison mode

Allows two locations or two data types within the same domain to be overlaid on the same chart for comparison.

5.5 Cycle annotations

Plan decisions and major field interventions are annotated on the temporal chart as vertical markers at their cycle position. This connects cause and effect — a steward can see a soil pH trend and trace it to the lime application decision made in Plan cycle 2.



Plan decision markers: blue. Show the objective title and selected option.

Act intervention markers: green. Show the task title and completion date.

Divergence markers: amber. Show the divergence type and description.

Markers are togglable — the steward can hide them to see the data trend without annotation noise.



5.6 Minimum data requirements for temporal display

A temporal chart requires a minimum of two data points at the same location and domain to be meaningful. Before this threshold is met:

The domain shows a single data point in the Domain Detail View with a note: ‘Timeline available after the next observation cycle.’

The ‘View timeline’ control is visible but disabled with this explanation on hover/tap.

This prevents empty or misleading charts from undermining confidence in the data.




6. Presentation mode



6.1 Purpose

Presentation mode is a clean, shareable view of selected Observe data for external stakeholders. It surfaces verified, timestamped evidence without exposing internal planning state, incomplete objectives, divergence flags, or unresolved Act tasks.



6.2 What is included and excluded

Included in Presentation mode

Excluded from Presentation mode

Verified data points with timestamps

Divergence records and unresolved flags

Photo evidence with GPS location and date

Act task status, proof slots, or submission state

Measurement trends from the Temporal Layer

Plan objectives, tier structure, or planning decisions

Species records and ecology observations

Locked or incomplete objectives

Before/after photo comparisons

Internal notes not intended for external review

Cycle-by-cycle trajectory charts

Any data from self-only tasks marked as internal

GPS-linked site maps showing observation locations

Steward names unless explicitly included



6.3 Presentation mode sections

Presentation mode organises Observe data into four sections suitable for external review:



Site overview — project name, location, area, date range covered, and total observation count. An optional project description field.

Current conditions — the most recent verified data point per domain, displayed as a clean domain-by-domain summary. Maps and photos prominent.

Ecological trajectory — temporal charts for domains where at least two cycles of data exist. Cycle annotations showing interventions and their outcomes.

Evidence library — a filterable gallery of all photos, GPS traces, and logged results. Filterable by domain, data type, and date range.



6.4 Sharing

Presentation mode generates a shareable link with configurable expiry (7 days, 30 days, 90 days, or permanent).

The link opens a read-only view in any browser — no OLOS account required.

The steward controls which domains are included in the shared view. Domains can be excluded individually.

A ‘Download PDF’ option generates a static report from the current Presentation mode view.

Shared links do not expose the OLOS project URL or any internal navigation.



Access control

Shared Presentation mode links are view-only and time-limited by default. They do not grant any access to Plan, Act, or internal Observe data beyond what the steward has explicitly included. Permanent links should be used only for certification or compliance purposes where a persistent record is required.




7. Observe and the Plan revision cycle



7.1 How Observe closes the spiral

Observe is the stage that makes Plan revision intelligent rather than arbitrary. Without Observe, Plan revision cycles are driven by schedule or by memory. With Observe, they are driven by evidence — the land tells the steward when conditions have changed enough to warrant a planning decision.



The Plan revision banner (section 3.5) is the primary mechanism. But Observe also passively informs Plan through the Domain Detail Views and Temporal Layer — a steward reviewing soil data before a planting decision is using Observe to ground that decision in current evidence.



7.2 Observe data as Plan inputs

When a Plan objective is opened in review mode (cyclical review — see Plan spec section 2.8), it surfaces the most recent Observe data for the domains it covers. This gives the steward the current evidence alongside the prior decision, making it straightforward to assess whether the decision still holds.



Plan objectives in review mode show a ‘Current Observe data’ panel alongside the existing decision record.

Observe data points that have changed significantly since the last Plan cycle are highlighted in amber.

The steward can navigate directly from a highlighted Observe data point to the relevant Domain Detail View or Temporal Layer.



7.3 Cycle tracking

Each Plan–Act–Observe cycle is a named, timestamped record. Cycle IDs are attached to every Act task and every Observe data point, creating a complete audit trail of when decisions were made and what evidence existed at the time.



Cycles are created automatically when a new Plan stage is initiated after a completed Act stage.

Cycles are named by default as ‘Cycle 1’, ‘Cycle 2’ etc. The steward can rename them (e.g. ‘Year 1 Establishment’, ‘Year 3 Intensification’).

The cycle indicator in the Observe top bar always shows the current cycle.

Archived cycle data is never deleted. The full history of observations and decisions is permanently accessible.




8. Developer acceptance criteria



8.1 Unified Land State

Default entry point for Observe. Shown when navigating to Observe without a specific domain context.

Domain cards reflect real-time data freshness state. A domain whose most recent data point passes its freshness threshold updates its badge without page reload.

Land health summary counts update immediately when new Act tasks are verified.

Plan revision banner appears and disappears based on the triggers defined in section 3.5. It is never shown when no triggers are active.

No health scores, ratings, or aggregated wellness indices rendered anywhere on the Unified Land State.

Plan revision banner renders at the correct priority level based on the trigger hierarchy in section 3.5. Critical triggers cannot be dismissed without a Plan action or explicit deferral. Informational triggers appear only in the pending reviews drawer, never as a full banner.

Multiple critical triggers are grouped into a single expandable banner, never stacked as separate banners.



8.2 Domain Detail Views

All GPS-linked data points render as pins or traces on the domain map.

Data points are ordered most recent first. Superseded data points are visually distinct (muted) but remain accessible.

Divergence records are displayed in a separate section and never mixed with verified task data.

Superseded data points are visible in the list with muted styling and a ‘Superseded’ badge. Superseding and superseded records link to each other bidirectionally.

The ‘Not a replacement’ control correctly restores both records to current status.

Domain-specific display components (section 4.4) render only when sufficient data exists to make them meaningful.



8.3 Temporal Layer

Temporal charts require a minimum of two data points at the same location before rendering. Below this threshold, the ‘View timeline’ control is visible but disabled with an explanatory message.

Cycle markers render at the correct temporal positions and are togglable without page reload.

Photo timeline strip is synchronised with the primary chart — the visible date range in the chart filters the photo strip.

Comparison mode supports exactly two overlaid series. A third selection replaces the oldest.



8.4 Presentation mode

Generates a shareable link with the configured expiry. Link opens in any browser without OLOS authentication.

Excluded data types (see section 6.2) are never accessible via the shared link, including through direct URL manipulation.

PDF export renders the current Presentation mode view as a static document. Charts rendered as images, not interactive.

Domain inclusion controls persist across sessions — a steward’s domain exclusion choices are remembered.



8.5 Data integrity

Only verified Act tasks update Observe data points. No other state triggers an update.

Divergence evidence enters Observe immediately on divergence capture submission, unconditionally.

Observe data is never edited or deleted from within the Observe surface. It is a permanent record. Corrections require a new verified Act task that supersedes the prior data point.

Cycle IDs are immutable once assigned. Renaming a cycle changes only the display name, not the ID.

This specification refines the Observe stage of OLOS into a developer-ready workflow. It keeps Observe focused on evidence gathering, site diagnosis, uncertainty, and constraint identification. It prevents Plan-stage decisions and Act-stage execution from leaking into Observe.

Observe records reality. Plan makes decisions. Act coordinates execution.

1. Executive Summary

OLOS is organized around three main stages: Observe, Plan, and Act. Each stage uses the same land-based Domains, but the user’s relationship to each Domain changes by stage. Observe is not for deciding what should happen or assigning work. Observe is for documenting what exists, what is happening, what is unknown, and what may constrain the project.

Developer takeaway

When the user selects Observe + Domain + Objective, the map view should load only the overlays and checklist needed for that specific observation task. The user should never be forced to see all land data at once.

9. Developer Acceptance Criteria

Acceptance Criterion

Requirement

Focused map loading

Selecting Observe + Domain + Objective loads only the default overlays required for that objective, with optional overlays available but collapsed.

Checklist-overlay linkage

Every checklist item should either create/edit a map feature, request evidence, populate a field, or confirm an observation.

Evidence gating

Objectives cannot be marked complete until required evidence fields are satisfied or intentionally marked Not Available with a reason.

No Plan leakage

Observe objectives must not require final design decisions, final prioritization, task assignment, or build scheduling.

Status outputs

Each objective must produce one or more status outputs: Clear, Unknown, Needs Investigation, Major Constraint, Potential Disqualifier.

Handoff packet

Completing an Observe objective creates a Plan handoff packet with summary, evidence, constraints, unknowns, and flagged questions.

Traceability

Every Plan-stage recommendation should be traceable back to Observe-stage evidence where possible.

Low cognitive load

The side panel should show only the active objective checklist, related evidence fields, and relevant map legend.

Implementation rule of thumb

Domain chooses the default lens. Objective chooses the working checklist. Checklist determines the required map actions and evidence. The map shows only what is necessary to complete the active objective.
