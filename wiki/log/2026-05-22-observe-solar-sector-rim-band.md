# 2026-05-22 — Observe: manual solar sectors render as compass rim band

**Branch.** `feat/atlas-permaculture`. Commit `5722ff31` (1 file, `SectorCompassDiagram.tsx`).

**Problem (steward report + reference image).** On the Observe map view, the Sector Compass HUD (`SectorCompassOverlay` → `SectorCompassDiagram`) drew a manually-added **solar** sector (Summer sun / Winter sun) as an **interior wedge that looked just like wind**. The steward wanted solar to **always** sit on the outer edge as a thin rim band — matching the auto-computed solar arcs — regardless of how the sector was added.

**Root cause.** The compass renders three layers: (1) computed **wind** petals (interior, frequency-scaled), (2) computed **solar** arcs (a thin band hugging the outer rim, `rOuter=R_OUTER`, `rInner=R_OUTER*0.82`), and (3) **manual** sector arrows from `externalForcesStore`. Layer 3 drew **every** manual sector as an interior wedge (`rOuter=R_OUTER*0.78`, `rInner=12`) with no special-casing — and `arrowToWedge` recast each one to `kind: 'wind-prevailing'`. So a manual `sun_summer`/`sun_winter` sector landed in Layer 3 and rendered exactly like wind.

**Fix (web-only, single file, type-keyed).** In `apps/web/src/v3/observe/modules/sectors-zones/SectorCompassDiagram.tsx`: added `const SOLAR_SECTOR_TYPES = new Set<string>(['sun_summer', 'sun_winter'])` and branched Layer 3 on `a.type` — solar types render with the **same rim-band geometry as the auto-computed Layer 2** (`rOuter={R_OUTER}`, `rInner={R_OUTER * 0.82}`, opacity 0.7); all other manual sectors keep the interior-wedge radii (`rOuter={R_OUTER * 0.78}`, `rInner={12}`). Removed the now-unused `manualWedges` memo. Existing manual solar **colors** are preserved (`#c4a265` summer, `#b87a3f` winter from `MANUAL_COLORS`) — only placement changed; the green wedge in the reference image illustrated *edge placement*, not a color change.

**Why type-keyed covers every entry path.** The steward flagged the toolbar's "Seed bearing from map" (`SunWindWedgeTool`) specifically but wanted the fix universal. Both the sector editor's `+ Add sector` and the toolbar tool persist the same `SectorArrow{type}` to `externalForcesStore`, and both render through the one `SectorCompassDiagram`. Keying on `type` (not on entry path) therefore fixes the editor, the toolbar, and the map-view HUD (`SectorCompassOverlay` reuses the diagram in compact mode) in one change.

**Verification.** Web tsc at the **3-error pre-existing baseline** (`StepBoundary.tsx`, two `HostUnion*` tests — unrelated, not from this change; `SectorCompassDiagram.tsx` clean). Live-app geometry verified deterministically by **DOM measurement** on the running web app (`getPointAtLength` on the rendered SVG paths): manual `sun_summer` (`#c4a265`, opacity 0.7) and `sun_winter` (`#b87a3f`, opacity 0.7) now measure radius **105→128** (rim band); manual `wind_prevailing` (`#5b7a8a`) and `noise` (`#7a6a9a`) stay **12→100** (interior) — fix confirmed, no regression to non-solar manual sectors. **`preview_screenshot` timed out repeatedly (30 s)** on this heavy MapLibre WebGL page even after isolating the SVG into a blank document — flagged per the preview-verification rule, **no visual screenshot claimed**; the DOM radius measurements are authoritative.
