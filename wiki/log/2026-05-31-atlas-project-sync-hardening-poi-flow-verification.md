# 2026-05-31 -- Project-sync hardening (commit e729365d) + POI resource-flow line live verification

**Branch.** `feat/atlas-permaculture` (single explicit-path commit `e729365d`, 7 files; rebased out-of-band, divergence-checked; **not pushed**). Closes the "stand up the wizard->API project-sync path so the serverId gate stops blocking cross-project features, then finish the live flow-line + idempotency verification" ask -- but a read-only live diagnosis **inverted the premise** before any code was written, so the work pivoted from "build the missing sync path" to "harden the existing one + finish the flow-line proof".

## Live diagnosis inverted the premise
Project->server sync **already works**. Of 17 local projects, 16 carry a `serverId`; `/auth/me` -> 200 (real `userId` + `defaultOrgId`), `/projects?status=all` -> 200 with 16 owned rows, every local `serverId` matched a real owned server row (`staleLocalServerIds` empty). The only rows failing the ownership gate are the public builtins (system-owned `00000000...` ids, plus the `mtc` demo with no `serverId`) -- which is correct: a steward cannot attach flows to a sample project they do not own. So there was no broken sync path to build. Per the steward's decision ("Add sync hardening first"), the session hardened the existing sync, then finished the live flow-line + layer-idempotency verification.

## Latent hazard the hardening fixed (double-create race)
`syncService.subscribeToProjects()` fired `syncProjectCreate(project)` on every *added* project -- un-awaited and NOT idempotent (no `serverId` short-circuit, no in-flight lock) -- WHILE both wizard create sites (`WizardStep1Site.tsx`, `NewProjectPage.tsx`) also called `api.projects.create()` inline. That was a double-create race (two server rows per wizard project). The fix routes every create through one idempotent canonical action.

## Phase 1 -- sync hardening (commit `e729365d`, 7 files)
- **1A -- canonical idempotent awaitable action.** New `export async function syncProjectNow(localId): Promise<{ok, serverId?, error?}>` in `apps/web/src/lib/syncService.ts`. Guards: not-found -> `{ok:false}`; `isBuiltin` -> `{ok:false, error:'builtin'}`; already-has-`serverId` -> `{ok:true, serverId}` short-circuit. Concurrent calls deduped via a module-level `inFlightProjectSync = new Map<string, Promise>()` (second caller for the same localId returns the in-flight promise). Body awaits `syncProjectCreate(project, /*rethrow*/ true)`, reads back the stamped `serverId`, resolves `{ok, serverId}`; `finally` clears the map entry. `syncProjectCreate` made idempotent at entry. `subscribeToProjects` now calls `void syncProjectNow(project.id)` so adds share the short-circuit + lock.
- **1B -- wizard awaits sync + surfaces failure.** `WizardStep1Site.tsx` and the non-template branch of `NewProjectPage.tsx` replaced the inline best-effort `api.projects.create()` with `const r = await syncProjectNow(project.id)` + a `toast.error` on `!r.ok`. Navigation STILL proceeds (local-first; sync failure only toasts, syncQueue retries). The `prefillTemplate` branch (its own server seam + showcase event) left untouched.
- **1C -- per-project sync-status indicator + manual "Sync now".** `PortfolioProjectList.tsx` (+`.module.css`) gains a per-row "Not synced" tag + "Sync" button for `!serverId && !isBuiltin` projects (local per-row pending state), a muted "Synced" affordance for synced non-builtins, "Sample" for builtins. `PortfolioMapPage.tsx` adds `handleSyncProject(id)` -> `await syncProjectNow(id)` -> `emitPortfolioToast` ('Project synced.' / "Sample projects can't be synced." on `error==='builtin'` / "Couldn't sync -- it'll retry automatically."), wired as `onSyncProject`.

## Phase 2 -- live flow-line + idempotency verification
DEV map handle (`if (import.meta.env.DEV) (window as ...).__portfolioMap = map;`, mirroring the existing `__ogdenProjectStore` handle) committed in `e729365d` to enable layer assertions. The verification then ran read-only against a live POI flow:
- **Retargeted the test project.** The original target **Halton Hills** could not be a flow endpoint: server-synced projects are created with `parcelBoundaryGeojson: null` (`syncService.ts:976`), so `projectCentroid(p)` (`portfolioModel.ts:166`) returns null -> no `centroidById` entry -> no line. Retargeted to **"Phase 4 Smoke"** (serverId `41b1a971-3803-4b76-9322-3ee417f17ca3`), which has BOTH a `serverId` AND inline boundary geojson, so it renders a centroid.
- **Exercised the flow.** Placed POI "Smoke Test Depot" near (-79.6, 43.5); created two flows -- `72633b2a` (compost, output, -> Halton Hills) and `37da95c0` (manure, input, <- Phase 4 Smoke). `POST .../portfolio-pois/:id/flows` -> **201** (twice); GET re-fetch persisted with `projectName` resolved.
- **Visual proof.** After reload + enabling the "Flows (N)" visibility toggle (the line is gated behind it, analogous to "Connections"), the **dashed manure line rendered Smoke Test Depot -> Phase 4 Smoke** (-86.5, 35.5), coloured by `materialKind`, dashed by direction. Direction routing correct: the `input` flow painted **only** on the `-input` layer. Screenshot captured (per [[project-screenshot-hang]] -- map page, no dead-API hang this time).
- **Idempotency.** Via `__portfolioMap.getStyle().layers`, each of `portfolio-poi-flow-line-output|-input|-bidir` existed **exactly once**; a basemap swap (topo-v2 -> streets-v2, fires `styledata`) left every count at 1 and the line survived.

## Discovered gap -- spun off as a follow-up task
Server-synced projects get `parcelBoundaryGeojson: null` and the Portfolio Map never hydrates it, so **server-only projects (e.g. Halton Hills) render no centroid and cannot be flow/relationship endpoints**. This is a boundary-hydration gap, not a flow-line defect. Spun off via a follow-up task (hydrate `parcelBoundaryGeojson` / fall back to `metadata.centerLat/centerLng` for server-synced projects) rather than expanded here.
