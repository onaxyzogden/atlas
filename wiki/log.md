- [2026-05-30 ‚Äî B4 tooltip max-height scroll cap (Slice K)](log/2026-05-30-b4-tooltip-scroll-cap.md) ‚Äî **Branch.** `feat/atlas-permaculture`. Closes Slice K of the B4 tooltip remaining-deferrals roadmap ‚Äî the highest-risk wave-C entry because **any** scroll affordance on the host-canopy-union tooltip requires `pointer-events: auto`, which collides with the 2026-05-25 invariant that the tooltip must NEVER steal the underlying union layer's mouseleave. **The carve-out.** `.tooltip[data-scrollable='true']` activates only when `pinned=true AND entries.length >= 4`. The CSS rule adds `max-height: calc(100vh - 80px)`, `overflow-y: auto`, and `pointer-events: auto`. Below either condition the invariant survives verbatim: hover mode + any host count = `pointer-events: none` and mouseleave-driven dismiss; pinned + 1‚Äì3 hosts = same. Above both conditions, mouseleave is moot ‚Äî when pinned, dismissal is via ESC, click-again-on-host, or tap-outside-canvas-and-outside-tooltip, none of which depend on the tooltip's pointer-events. **Tap-outside handler carve-out (PlanDataLayers).** `onDocPointerDown` would have dismissed pointerdowns on the tooltip's scrollbar/content (target not in canvas ‚Üí fall through). New early-return: `if (target instanceof Element && target.closest('[data-testid="host-canopy-union-tooltip"]')) return;`. The tooltip is portalled to the document body so PlanDataLayers can't hold a ref to it ‚Äî the data-testid already used by tests becomes the production query selector, no portal-aware ref forwarding needed. **Edge-clamp adjustment.** Raw `entries.length * PER_BLOCK_H` would over-reserve vertical space when the surface is scroll-capped (e.g. 20-host pinned stack) and push the anchor unnecessarily high; new `scrollCapMaxH = Math.max(PER_BLOCK_H, viewportH - 80)` bounds the estimatedH when `scrollable`. `Math.max(PER_BLOCK_H, ...)` guards against pathologically short viewports. **Threshold = 4.** Three hosts (~340px) fits 768px viewport comfortably; four (~448px) tight; five+ clips. Single `SCROLL_CAP_THRESHOLD = 4` module constant so a future tuning round can move it without touching consumers. **Reserve = 80px.** `calc(100vh - 80px)` matches the existing edge-clamp's intuitive budget (12px cursor gap + 8px padding at top/bottom + DesignToolRail headroom); on a 720px viewport ~640px usable ‚âà 5.9 blocks, "scroll if you must" not "scroll constantly." **Why not WAAPI / wheel-capture.** The roadmap's alternative ‚Äî capture `wheel` events on the document, translate to a CSS-variable offset on a virtual scrollable container ‚Äî would preserve the 2026-05-25 invariant verbatim but re-implement overscroll, inertia, scrollbar-drag, keyboard scroll, and require a separate `touchmove` path for tablets. The dual carve-out (CSS + tap-outside) ships in ~10 lines and inherits every native scroll affordance for free; the narrow carve-out window keeps the invariant holding in every state the steward meets during normal hover flow. **Why threshold-by-entry-count not viewport-bound.** The cleaner predicate would be `scrollable = pinned && rawEstimatedH > scrollCapMaxH`, but (a) entries.length is stable across viewport changes (resize doesn't toggle the carve-out under a pinned steward) and (b) the test surface stays simpler. If the fixed threshold proves wrong for small viewports, swap the predicate without changing the data-attribute contract. **Reused, not built:** existing portal mount; existing data-testid; existing tap-outside listener; existing edge-clamp logic; existing fade machinery (data-visible / data-exiting / transition tokens). **Tests added:** 4 under `scroll-cap carve-out (Slice K)` describe ‚Äî below-threshold pinned = no data-scrollable; above-threshold unpinned = no data-scrollable (hover invariant survives); at-threshold pinned = data-scrollable='true'; very-tall (20-host) pinned at viewport-top = data-scrollable='true' + edge-clamp produces non-negative top (regression guard). **Verification:** `npx vitest run src/v3/plan/layers/__tests__/HostCanopyUnionTooltip.test.tsx` ‚Üí 12/12 (8 prior + 4 new); `npx vitest run src/v3/plan src/features/agroforestry` ‚Üí 278/278 (35 files; +4 over Slice J); `npx tsc --noEmit` clean on the three touched files (pre-existing unrelated errors elsewhere unchanged); preview-server visual check not possible in this worktree per project CLAUDE.md "say so rather than assuming success" ‚Äî data-attribute contract fully unit-test-covered, visible scroll relies on browser-native `overflow-y: auto`. **Invariants preserved:** 2026-05-25 (mouseleave-driven dismiss in hover mode + pinned-small-stack), 2026-05-26 (single-pin), 2026-05-27 (multi-feature fan-out), 2026-05-28 (touch tap-outside dismiss), 2026-05-29 + 30 (enter + exit + per-block + reverse-in-flight fade), `prefers-reduced-motion`. **Out of scope:** Slice L multi-pin (separate slice; depends on this one), Slice M hover-card drill-down (depends on L), viewport-aware predicate, sticky block headers when scrolling, keyboard arrow-key navigation between blocks. - [2026-05-30 ‚Äî B4 motion-token harmonisation (Slice I)](log/2026-05-30-b4-motion-token-harmonisation.md) ‚Äî **Branch.** `feat/atlas-permaculture`. Closes Slice I of the B4 tooltip remaining-deferrals roadmap. Extracts the four hard-coded overlay-fade literals from `HostCanopyUnionTooltip.module.css` (duration `120ms`, translate `2px`, easing `ease-out`/`ease-in`) into a new `--motion-overlay-*` token family declared on `:root` in [apps/web/src/styles/tokens.css](../apps/web/src/styles/tokens.css) alongside the existing `--duration-*` / `--ease-*` families. The 200ms safety timeout in `PlanDataLayers.tsx` does duplicate `--motion-overlay-duration + slack`, but the slack is the load-bearing part ‚Äî if a second JS consumer materialises we'll revisit. **Why these four tokens specifically.** Two easings (not one) because enter accelerates toward the visible state and exit decelerates away from it; CSS variables can't carry direction. - [2026-05-30 ‚Äî B4 follow-up: per-block tooltip fade + reverse-in-flight](log/2026-05-30-b4-tooltip-perblock-fade-and-reverse.md) ‚Äî **Branch.** `feat/atlas-permaculture`. Transitions get the same behaviour with zero JS: they always interpolate from the current computed value, which is exactly what "reverse" means for a single-property fade. **Why merge by hostId.** When the active set shrinks, the dropped host has to stay in the displayed array long enough to fade; indices shift under set changes, so a key-by-index would cancel its transition when the array re-reconciled.
