# 2026-05-11 — deck.gl ScenegraphLayer migration + 3D feature coverage

## Context

Atlas's Plan and Observe canvases rendered placed Built-Environment
entities as 3D using a bespoke three.js `CustomLayerInterface` —
`apps/web/src/v3/builtEnvironment/layers/DesignElementGlbLayer.tsx` —
plus a MapLibre `fill-extrusion` fallback in
`DesignElementExtrusionLayer.tsx`. The pipeline worked but:

- All 14 registered structure kinds shared one generic `_generic_box.glb`.
- Vegetation, earthworks, and zone markers were intentionally
  *excluded* from 3D — they had no kind entries at all.
- The custom three.js layer required hand-management of GL state and
  was not reusable for other 3D effects (instancing, picking, etc.).

The briefing at `Downloads/OGDEN_LandOS_3D_Models_Briefing.md` proposed
adopting deck.gl's `ScenegraphLayer` via `@deck.gl/mapbox`
`MapboxOverlay`, which gives us per-instance scenegraph URLs,
PBR lighting, picking, and instanced rendering for free — at the cost
of ~400 KB gzipped of new dependency (deck.gl v9 + luma.gl v9).

## Decision

Migrate the entire 3D model pipeline to deck.gl `ScenegraphLayer`,
retire the bespoke three.js layer, and expand kind coverage to four
new categories. The user explicitly chose this path (over keeping the
three.js layer and just authoring more GLBs).

### What landed (Phases 1–5)

| Phase | Outcome |
|---|---|
| **P1 — pipeline parity** | New `DesignElementScenegraphLayer.tsx` mounted under a `<DeckOverlay>` wrapper around `MapboxOverlay`. The three.js `DesignElementGlbLayer.tsx` was deleted. Both Plan and Observe canvases mount the deck overlay. |
| **P2 — authored structure GLBs** | All 14 structure kinds got per-kind procedural GLBs authored by `scripts/gen-structure-glbs.mjs`. No more shared grey box for them. |
| **P3 — vegetation** | 5 new kinds (oak-tree, pine-tree, apple-tree, shrub, hedgerow) wired through `BUILT_ENVIRONMENT_KINDS` + `elementCatalog` + `elementHeights`. 4 procedural GLBs in `public/models/vegetation/`; hedgerow stays flat (line geometry). |
| **P4 — earthworks + zone markers** | 9 new kinds: berm, raised-bed, terrace + zone-0…zone-5. Procedural GLBs in `public/models/earthworks/` and `public/models/zone-markers/`. |
| **P5 — per-instance rotate / scale + picking** | `BuiltEnvironmentEntity.proposed` now carries optional `rotationDeg` + `scaleMul`. `ScenegraphLayer` reads them per instance. `onClick` opens the existing inline-edit popover via `openBeInlineEditById`. Generic BE edit schema gained rotation + scale fields for GLB kinds. |

### P6 — Custom GLB upload (shipped 2026-05-11 follow-up)

The IndexedDB-backed user-upload flow shipped after the main migration
commit. Final shape:

- `customModelStore.ts` — IDB `ogden-custom-models/blobs` for binary;
  catalog metadata in `localStorage` under `ogden:custom-models:catalog`;
  `URL.createObjectURL` regenerated on hydrate per entry.
- `customModelValidator.ts` — magic-byte + version-2 + JSON-chunk check,
  KHR extensions-used/required intersected with a 12-entry allowlist,
  ≤10 MB, SHA-256 fingerprint via `crypto.subtle`. Full structural parse
  deferred to luma.gl's glTF loader at render (which falls back to the
  generic box on failure).
- `CustomModelPalette.tsx` — floating bottom-right card in
  `VisionLayoutCanvas`. Tile click arms the canonical
  `plan.structures-subsystems.be.custom-glb` tool and stashes the
  active modelId in `customDrawSelectionStore`.
- `BeV2ExistingTool` — on draw-complete, if kind is `custom-glb` it reads
  the stash and stamps `proposed.customModelId` on the new entity.
- `DesignElementScenegraphLayer` — at render time, if entity is
  `custom-glb` and `customModelId` resolves in the store, swap
  `spec.glbUrl` for the per-instance blob URL.

Architectural call: one canonical `custom-glb` kind in
`BUILT_ENVIRONMENT_KINDS` rather than synthetic `custom:<id>` kinds
per-upload. Per-instance modelId rides `proposed.customModelId` so the
registry, store filters, and edit-schema builders stay finite. The kind
is filtered out of `BE_TOOL_ITEMS` so it never surfaces as a bare rail
button — placement requires a selected uploaded model.

### Procedural over CC0 acquisition

The plan called for downloading Kenney/Quaternius CC0 GLBs. In practice,
binary fetches were unreliable from this environment, so we authored
all GLBs procedurally via `scripts/gen-*-glbs.mjs` against a shared
`scripts/lib/glb-writer.mjs` + `scripts/lib/primitives.mjs`. This:

- Keeps assets CC0 (generated by code in this repo).
- Makes adding kinds a one-file edit + `pnpm gen:models` run.
- Avoids supply-chain pinning of third-party asset packs.

Cost: silhouettes are symbolic, not photoreal. Acceptable for v1; a
future session can swap individual GLBs for authored art without
changing any wiring.

### Built-environment registry naming

Despite "Built Environment" not strictly describing trees, we kept
`BUILT_ENVIRONMENT_KINDS` as the single registry and added new
categories (`vegetation`, `earthworks`, `zone-marker`). Forking a
sibling registry would 3–5× the per-kind code without adding
capability — the renderer pipeline, store, and inline-edit forms
don't distinguish.

## Consequences

- Bundle size up ~400 KB gzipped (deck.gl + luma.gl).
- Pipeline now supports instancing, picking, PBR for free.
- 14 structures + 4 vegetation + 3 earthworks + 6 zone markers
  render with distinct authored silhouettes.
- `BuiltEnvironmentEntity.proposed.scaleMul` is new; entities created
  before this session simply default to `1.0` via the optional read.
  No persist-store version bump was needed (additive optional field).
- Custom GLB upload shipped — floating palette in Vision Layout canvas
  accepts user `.glb` files, validates them, persists blobs to
  IndexedDB, and places per-instance models via the canonical
  `custom-glb` kind.
