## Context

Phase E closed the **Apricot-Lane Validation Protocol at 4/4** (Data
Layering, Hydrology, Enterprise Stacking, Natural Capital). The Phase E
ADR explicitly deferred four items as "post-protocol aspirations" —
**none gating the protocol score**, but all valuable enough to warrant
their own slice:

1. **Live Playwright Anti-GIS screenshot.** Phase E verified the
   code-path that produces the mobile Verdict-above-the-fold view, but
   the screenshot itself was generated by hand. We wanted a programmatic
   driver as durable evidence.
2. **`evidence_audit` server-side persistence.** The Evidence selector
   framework emits fragments deterministically from selector inputs, but
   nothing was anchoring those emissions in a reproducibility ledger.
3. **Per-zone SOM trajectory.** `som_trajectory_yearly` (migration 031)
   already keyed on `(project_id, zone_id, year)`, but `projectSomTrajectory`
   only emitted whole-project rows (`zone_id = NULL`).
4. **LAUNCH-CHECKLIST transparency.** None of the four items above were
   recorded as either blockers or accepted aspirations — opaque to a
   reader auditing pre-commercial readiness.

Phase F lands all four behind one commit per sub-phase, with this ADR +
the companion log filed per-phase (not per sub-phase). The covenant
restated in Phase E continues to apply verbatim: **appreciation of
stewarded land value, not investor yield**, per
[[fiqh-csra-erased-2026-05-04]]. None of the F changes touch capital
framing; they add observability and reproducibility on top of the
covenant-compliant surfaces that Phase E delivered.

## Decisions

### F.1 — Playwright Anti-GIS programmatic snapshot

**Decision.** Add `apps/web/scripts/anti-gis-snapshot.ts` as a
**programmatic** Playwright driver (no `playwright.config.ts`,
no `@playwright/test` runner). The pattern mirrors the existing
`scripts/snapshot-scene-images.ts` — `chromium.launch({ headless: true })`,
`newContext({ viewport: { width: 390, height: 844 }, deviceScaleFactor: 2 })`,
navigate to `localhost:5200/v3/project/${APRICOT_LANE_PROJECT_ID}/observe`,
wait `networkidle` + a 2.5 s settle, then assert that the Verdict card
heading **and** the first Next Best Action heading both have
`boundingBox.y + height <= viewport.height` (above the fold). PNG is
written to `apps/web/screenshots/anti-gis-apricot-lane.png` (gitignored
via `apps/web/screenshots/*.png`; the directory is tracked via
`.gitkeep`). Exposed as `pnpm --filter @ogden/web run anti-gis:snapshot`.

**Rationale.** Playwright 1.60.0 is already a `@ogden/web` devDependency
(used by `prerender:showcase` and `scripts/snapshot-scene-images.ts`).
Adding a config file + `playwright.config.ts` + `npx playwright install`
would have introduced a parallel test harness with its own discovery
rules — overkill for a single screenshot. The programmatic pattern
keeps the driver in lockstep with the existing snapshot tooling and
inherits the same `tsx` runner.

**Optional auth.** The script honours `ANTI_GIS_STORAGE_STATE` (path
to a Playwright `storageState.json`) so a future CI integration with a
seeded authenticated session can drop in without code changes. F.1 itself
runs against the unauthenticated public route; the storageState seam is
inert until set.

**Alternatives considered.**
- **Full `@playwright/test` runner.** Rejected: imports a parallel test
  framework alongside Vitest, fragments the test story. The single
  screenshot is not worth the surface.
- **Puppeteer.** Rejected: would add a dependency that duplicates
  Playwright's coverage. Playwright is already in the tree.

### F.2 — LAUNCH-CHECKLIST Post-Protocol Aspirations

**Decision.** Append a new section to `wiki/LAUNCH-CHECKLIST.md`
titled **"Post-Protocol Aspirations (Phase E + F follow-ups)"**, kept
distinct from the existing Legal / Operational / Compliance blocks.
Items are split into "Landed in Phase F" (F.1, F.3, F.4) and
"Deferred (post-Phase-F)" (replay tool, remaining-panel rollout for
the audit hook, per-zone web overlays, tooltip Evidence retrofit, PDF
Evidence surface, i18n, per-fragment confidence ranges, Playwright CI
integration).

**Rationale.** These items are **not pre-commercial blockers** — Atlas
passed the Apricot-Lane protocol at 4/4 in Phase E. Putting them under
"Legal / Operational / Compliance" would have falsely conflated them
with the live GAEZ NC-license blocker. The new section signals: "we
know about these, we're transparent, none of them stop a commercial
launch."

### F.3 — Per-zone SOM trajectory

**Decision.** Extend the soil-regeneration route to accept an optional
`zones[]` payload on POST and an optional `?zoneId=<id>` query on GET.
Keep `projectSomTrajectory` (the v1 producer) **unchanged**; add a
**new** producer `projectSomTrajectoryWithZones` returning
`{ projectRows, zoneRows }`. The route handler calls the new producer
and DELETE-then-INSERTs both layers in a single transaction. The
upsert key for whole-project rows is `(project_id, zone_id IS NULL,
year)` (Postgres treats NULL as distinct in UNIQUE, so DELETE-then-
INSERT is required); per-zone rows use the normal `(project_id, zone_id,
year)` UNIQUE behaviour.

**Rationale.** The plan called for "extending `projectSomTrajectory`
with optional `zones[]`", which would have changed the function's
return type and forced three downstream call-sites
(EconomicsPanel, ScenarioPanel, the D.3 service test) to migrate.
Keeping v1 intact and adding a parallel function honours
[[feedback-no-deletion]] — the v1 producer remains useful, and the
blast radius of F.3 is one route file + the new function. Web consumers
of the chart and scenario panels remain on v1 in F.3 and only need to
migrate when they actually want per-zone overlays (deferred).

**Schema.** Migration 031 already supported `(project_id, zone_id, year)`
UNIQUE with `zone_id TEXT NULLable`. **No new migration.**

**Web surfaces unchanged in F.3.** `JCurveChart`, `somAppreciation`,
`EconomicsPanel`, `ScenarioPanel` continue to consume whole-project
rows. A future slice can surface per-zone overlays.

### F.4 — `evidence_audit_log` + LandVerdictCard emit hook

**Decision.** New migration **033** creates `evidence_audit_log`:
- `project_id` (FK projects, ON DELETE CASCADE)
- `panel_key` (TEXT) — stable identifier for the emitting surface
  (`LandVerdictCard` in F.4)
- `input_hash` (`CHAR(64)`) — SHA-256 hex of the stable-stringified
  selector inputs
- `input_payload` (JSONB) — the inputs themselves (verbatim) so the
  replay tool can recompute and assert byte-identical outputs
- `selector_name` (TEXT) — e.g. `selectEvidenceFor(land-verdict)`
- `evidence_output` (JSONB) — the emitted Evidence object
- `created_at`, `created_by`

Indexes on `(project_id, panel_key, created_at DESC)` and on
`input_hash` (the reproducibility join key).

New route `POST /api/v1/projects/:projectId/evidence-audit/log`
(preHandlers: `[authenticate, resolveProjectRole]`, any project member,
write-only — no GET in v1). Mounted in `app.ts` adjacent to
soil-regeneration.

Web side:
- `lib/evidence/hashInputs.ts` — `stableStringify` (recursive key
  sort) + `hashInputs` (SHA-256 via `crypto.subtle.digest`).
- `lib/evidence/auditEmit.ts` — `emitEvidenceAudit(...)` fire-and-
  forget helper. Errors are swallowed; callers never await; render is
  never blocked.
- `apiClient.ts` — `api.evidenceAudit.log(...)` binding.
- **`LandVerdictCard.tsx`** is the only initial adopter. Inputs to
  `selectEvidenceFor(...)` are split into a memoised object so a
  `useEffect` keyed on that memo can call `emitEvidenceAudit(...)`
  once per distinct input set per mount.

**Rationale for `LandVerdictCard`-only adoption.** The card is the
single most-load-bearing Evidence-emitting surface (gates the
Anti-GIS Verdict + first Next Best Action — the entire 30-second
mobile glance). If anything goes wrong (perf, hash collisions, write
floods), we want to see it on the highest-traffic surface first
before we roll out to the other seven panels (DecisionTriad,
IntelligenceSummaryCard, SiteSummaryNarrativeSection, WaterStorageCard,
ThreeEthicsRollupCard, WaterRouterCard, CapitalPartnerSummaryExport).
The rollout is tracked under LAUNCH-CHECKLIST Post-Protocol
Aspirations and remains explicitly deferred.

**No GET in v1.** The audit log is a passive ledger. The intended
consumer is a deferred **server-side replay tool**: given an
`input_hash`, look up the row, recompute `selectEvidenceFor` with the
stored inputs, and assert byte-identical output. Until that exists,
exposing a GET would invite premature client-side coupling.

## Covenant restatement

Atlas's capital framing remains: charitable donation, restricted
donation, qarḍ ḥasan (interest-free loan), in-kind contribution,
sponsorship. **No CSRA. No salam-style advance-purchase.** Public-
facing label for capital contributors remains "capital partners &
allies." The Phase F changes are all observability / reproducibility
plumbing — they neither widen nor narrow the covenant surface
established in [[fiqh-csra-erased-2026-05-04]] and ratified in Phase E.
