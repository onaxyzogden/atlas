OLOS Project Type Templates Developer Specification

OLOS Project Type Templates

Developer specification for Stage Zero configuration, domain priorities, map overlays, starter objectives, and stage handoffs

Core Product Rule

Project Type Templates are Stage Zero presets. They do not replace Observe, Plan, or Act. They configure which domains, overlays, questions, objectives, warnings, dashboards, and handoff packages OLOS should emphasize for a particular land-based project.

1. Purpose

This document defines reusable project type templates for OLOS. When a user chooses a land-based project type in Stage Zero, the selected template should preload a guided workflow across Observe, Plan, and Act. The goal is to prevent the user from seeing every possible domain and overlay at once while still preserving the universal OLOS architecture.

A project type template should behave like a configuration layer: it weights domains, chooses default overlays, asks relevant setup questions, loads starter objectives, checks for red flags, and shapes dashboards. Users should still be able to customize, combine, or override templates.

2. How Templates Fit the OLOS Architecture

Layer

Role in OLOS

Stage Zero

Defines the project type, intent, constraints, non-negotiables, expected outputs, and initial feasibility questions.

Project Type Template

Preloads the domain priority model, relevant overlays, starter objectives, dashboards, and risk checks.

Stages

Observe gathers evidence, Plan makes decisions, and Act executes, verifies, maintains, and feeds learning back into the system.

Domains

Universal areas of land intelligence that remain consistent across project types.

Objectives

Focused pieces of work inside a domain. Objectives determine the checklist and map view state.

Overlays

Map lenses loaded only when useful for the selected domain + objective.

Handoffs

Structured outputs passed from Stage Zero to Observe, Observe to Plan, and Plan to Act.



Developer Implementation Note

The template should not hard-code a single path. It should create a recommended path with editable defaults. A user may start with Homestead, add a Market Garden component, then later activate Agritourism or Education objectives.

8. Detailed Project Type Templates

1. Homestead / Family Land Template

For a household-scale land project focused on food security, simple infrastructure, daily living, resilience, and manageable stewardship rather than maximum commercial production.

Field

Template Specification

Template ID

PT-HOMESTEAD

Best fit

One household or family stewarding land for personal food, animals, shelter, water, energy, and long-term resilience.

Default success outcome

A realistic household land plan with safe access, water, food systems, basic infrastructure, and maintenance rhythms sized to the household capacity.

Primary domains

Vision & Project Intent; Land Base & Legal Context; Hydrology & Water; Soil & Subsurface; Plants, Crops & Food Systems; Animals, Livestock & Wildlife; Built Infrastructure; Energy, Materials & Resource Flows; Economics & Capacity; Monitoring, Records & Feedback.

Supporting domains

Climate & Microclimate; Topography & Landform; Ecology & Biodiversity; Access, Circulation & Logistics; People, Roles & Governance; Risk, Compliance & Suitability.

Default overlays

Zones; Water Flow; Soil Conditions; Access / Movement; Infrastructure / Utilities; Suitability; Timeline / Phasing; Stewardship Intensity; Risk / Compliance.

Stage Zero questions to ask

Primary household goals; desired food self-reliance percentage; animals desired; residence status; off-grid intensity; available weekly labour; budget range; tolerance for temporary infrastructure; public/guest access level.

Observe starter objectives

Verify dwelling/legal constraints; map water access and runoff; document existing food-growing areas; assess soil near daily-use zones; map current access, storage, and utility points; document animal/wildlife pressure.

Plan starter objectives

Choose Zone 0/1/2 layout; select first-year garden and animal areas; decide water storage/irrigation approach; phase infrastructure by budget and labour; set maintenance schedule.

Act starter objectives

Assign setup tasks for access, water, garden, animal shelter, compost, storage, and safety; verify completion with photos, receipts, and recurring checklists.

Red-flag / disqualifier checks

No legal dwelling path; unreliable water; severe access limitations; animal bylaws blocking desired livestock; excessive maintenance burden for available labour; flood/fire/contamination issues near living areas.

Recommended dashboard widgets

Homestead readiness score; household stewardship load; water/food/infrastructure dashboard; weekly chore rhythm; first-year buildout checklist.

Developer notes

Default to small, practical bundles. The system should prevent overbuilding and compare every proposed feature against household labour capacity.

Field

Template Specification

Template ID

PT-REGEN-FARM

Best fit

Commercial or semi-commercial farms with multiple enterprises such as pasture-raised poultry, market garden, grazing, orchards, compost, education, or CSA.

Default success outcome

A sequenced farm development plan that matches land capability, soil/water realities, enterprise economics, labour capacity, and ecological regeneration targets.

Primary domains

Hydrology & Water; Soil & Subsurface; Ecology & Biodiversity; Plants, Crops & Food Systems; Animals, Livestock & Wildlife; Access, Circulation & Logistics; Built Infrastructure; Economics & Capacity; Monitoring, Records & Feedback.

Supporting domains

Vision & Project Intent; Land Base & Legal Context; Climate & Microclimate; Topography & Landform; Energy, Materials & Resource Flows; People, Roles & Governance; Risk, Compliance & Suitability.

Default overlays

Zones; Sectors; Contours / Landform; Water Flow; Soil Conditions; Ecology / Habitat; Access / Movement; Infrastructure / Utilities; Suitability; Timeline / Phasing; Stewardship Intensity; Monitoring / Records.

Stage Zero questions to ask

Enterprise mix; commercial intensity; livestock species; crop systems; processing/wash/packing needs; market channels; labour model; equipment access; water demand; soil regeneration goals; certification/compliance needs.

Observe starter objectives

Map hydrology and slope; complete soil baseline; identify production-suitable areas; document existing infrastructure and access; map livestock/wildlife pressure; record current budgets, labour, and equipment.

Plan starter objectives

Select enterprise zones; phase water, fencing, access, soil improvement, crop/livestock layout, and wash/packing infrastructure; build enterprise budgets and capacity checks.

Act starter objectives

Launch enterprise work packages; assign rotations, planting, infrastructure builds, procurement, maintenance, monitoring, and harvest records.

Red-flag / disqualifier checks

Water demand exceeds supply; poor access for equipment/markets; zoning blocks commercial farm activity; high maintenance intensity with low labour; soil remediation too costly; no proof path for claimed regenerative outcomes.

Recommended dashboard widgets

Enterprise map; soil/water improvement tracker; rotation calendar; budget vs capacity dashboard; production records; proof-of-regeneration log.

Developer notes

This is one of the highest-complexity templates. The UI should rely heavily on phasing and capacity gates before allowing a full buildout plan.

Default success outcome

A compact production layout with strong soil, reliable water, efficient access, practical crop blocks, harvest flow, and realistic labour/cash planning.

Primary domains

Observe starter objectives

Recommended dashboard widgets

Crop block map; irrigation checklist; weekly planting/harvest calendar; labour forecast; gross margin by crop block; soil amendment log.

Developer notes

Default map should stay tight around active production areas. Avoid loading whole-property complexity unless selected by the user.

Field

Template Specification

Template ID

PT-FOOD-FOREST

Best fit

Default success outcome

A phased perennial layout matched to soil, water, microclimate, access, stewardship capacity, and long-term establishment needs.

Primary domains

Climate & Microclimate; Topography & Landform; Hydrology & Water; Soil & Subsurface; Ecology & Biodiversity; Plants, Crops & Food Systems; Monitoring, Records & Feedback.

Supporting domains

Default overlays

Zones; Sectors; Contours / Landform; Water Flow; Soil Conditions; Ecology / Habitat; Suitability; Timeline / Phasing; Stewardship Intensity; Monitoring / Records.

Primary outputs; species preferences; orchard vs food forest structure; water availability; wildlife pressure; maintenance capacity; establishment timeline; propagation strategy; harvest/use model.

Observe starter objectives

Map sun, wind, frost, slope, water movement, soil quality, existing trees, wildlife browsing, invasive pressure, and access for establishment and harvest.

Plan starter objectives

Select perennial zones; choose species/guilds; phase canopy/shrub/herbaceous layers; design irrigation/mulch/protection; plan wildlife protection and maintenance intervals.

Act starter objectives

Recommended dashboard widgets

Species suitability matrix; guild builder; planting phasing timeline; survival tracker; seasonal maintenance checklist; browse pressure map.

Developer notes

The template must emphasize time. Perennial systems need Year 0/1/3/full-vision phasing and should distinguish establishment care from mature stewardship.

Developer notes

Treat animals as moving forces. Every task should connect location, timing, responsible steward, and welfare verification.

A user can select one primary project type and optional secondary project types during Stage Zero.

Selecting a template changes domain priority, default overlays, starter objectives, red-flag checks, and dashboard widgets.

The map view loads overlays based on selected domain + objective, then modified by project type context.

Each template creates at least one starter objective in Observe, Plan, and Act.

Hybrid templates do not duplicate objectives; they merge objective packs and warn when capacity exceeds available labour/budget.

Templates can be edited, disabled, or reweighted by the user or administrator.

Every template includes risk/compliance checks appropriate to its use case.

Every template feeds a clean handoff into the next stage rather than trapping information inside a static setup form.

Project type templates should make OLOS feel like it understands the user’s land-based goal from the beginning. The user should not have to assemble the entire operating system manually. The template should load the right starting path, then let the land evidence refine the path over time.

1. Purpose & architectural context

OLOS organises land stewardship work around a universal 7-tier Plan stage. The tier structure is fixed — it reflects the epistemic dependency chain of regenerative land design. What varies by project is which objectives are loaded within each tier.



The Project Type & Secondary Layer system is the configuration mechanism that determines which objectives populate the universal tier structure for a given project. It operates on three layers: a universal objective set present for every project regardless of type; a primary-type objective set that anchors the project's core logic; and secondary objective sets that expand the project via additive or modifying behaviour.



Architectural principle  The tier structure encodes how land design must proceed. Project type encodes what kind of land project is being designed. These are separate concerns and must remain so in the data model.

7. Modifying secondary mechanism

7.1 The patching model

When a modifying secondary is applied to a project, it expands existing primary objectives by injecting additional checklist items into them. This is the patch record model. It keeps the objective unified — the steward works one checklist, not two — while ensuring the expanded scope is explicitly represented and traceable to its source.



7.2 Patch record structure

Each modifying secondary carries one or more patch records. Each patch record targets a single objective and defines the injection precisely.



Field

Purpose

target_objective_id

References the existing objective that receives the patch

inject_after_item_id

Placement anchor within the target checklist. Null = append after last native item

items[]

The checklist items being injected. Each item carries: id, text, is_methodology boolean, feeds_into string

source_secondary_id

Tags each injected item to its originating secondary. Required for removal tracking and UI labelling

scope_note

Plain-language description of what the secondary adds to this objective. Displayed in the objective detail panel as: 'Scope expanded by: [Secondary name]'

completion_gate_amendment

Optional. Updated completion gate text for when the secondary changes what 'done' means for the objective. Example: water strategy completion gate expands from 'water strategy approved' to 'water strategy approved including stock water system' when Silvopasture is added



7.3 Conflict resolution — multiple secondaries patching the same objective

When two or more secondaries both carry patch records targeting the same objective, their injections are applied in the order the secondaries were added to the project. Injected items from each secondary append after the last native item (or after the previous secondary's injections if multiple). Each injected item retains its source_secondary_id tag regardless of ordering.



7.4 Reopening completed objectives

If a secondary is added after an objective has been marked complete, and that secondary's patch records inject items into that objective, the objective reverts to active status. Completed checklist items remain checked. Only the newly injected items appear as outstanding.



UX requirement  This reopening must never be silent. A named modal alert is required — never a badge or a silent state change. The alert must name the secondary being added, list each affected objective, state how many new checklist items have been added, and provide a direct link to each affected objective. The steward actively acknowledges before continuing.



7.5 Completion gate amendments

Some modifying secondaries change not just what work is required, but what 'done' means for the objective. When a patch record carries a completion_gate_amendment field, the target objective's completion gate text is updated to the amended version. The previous gate text is preserved in the objective's version history. Both texts are shown in the detail panel: current gate (active) and previous gate (greyed, with amendment source noted).



7.6 Scope note display

The objective detail panel must display a scope note for any objective that has been expanded by one or more modifying secondaries. Format: 'Scope expanded by: [Secondary name 1], [Secondary name 2]'. This is shown below the objective description and above the checklist. It links directly to the secondary's project recipe entry.






8. Mid-project secondary addition

8.1 Design rationale

Land projects reveal themselves over time. Blocking secondary addition at project creation fights the actual rhythm of land stewardship. OLOS must accommodate evolving project scope without requiring a project restart.



8.2 Delta computation protocol



Step

Behaviour

1. Audit completed work

Identify all objectives already complete or active. These are never re-gated.

2. Compute delta objectives

Identify objectives the new secondary introduces that do not already exist in the project.

3. Insert delta at correct tier

New objectives are placed in the appropriate tier per the universal tier structure. No new tiers are created.

4. Apply patch records

If the secondary is modifying on this primary, apply all patch records to their target objectives. If any target is complete, trigger the reopening alert.

5. Flag Observe gaps

If the secondary requires Observe data not yet collected, surface a named gap notification. Never fail silently.

6. Flag design tensions

If the secondary creates a tension with the existing primary or any active secondary, surface the tension warning immediately.



8.3 Secondary removal

Removing a secondary mid-project is permitted only if none of its delta objectives are active or complete. If any delta objective has been started or finished, removal is blocked. The UI must name which objective(s) are preventing removal and offer the option to mark them as Deferred instead.



8.4 Version history

Every change to the project type configuration — initial selection, secondary addition, secondary removal, tension acknowledgement, reopening acknowledgement — is recorded as an append-only versioned entry with action type, timestamp, and actor. This record is never overwritten.

created_at — timestamp of initial selection

tension_acknowledgements — array of { tension_id, acknowledged_at, acknowledged_by }

reopening_acknowledgements — array of { secondary_id, affected_objective_ids[], acknowledged_at, acknowledged_by }



9.2 Objective record (relevant fields)

objective_id

tier — integer 0–6

source — enum: universal | primary | secondary

source_type_id — null if universal; references project type catalogue if primary or secondary

secondary_class — null if not a secondary objective; enum: additive | modifying

patch_records[] — array of patch records this objective carries when acting as a modifying secondary source

injected_items[] — checklist items injected from secondary patch records, each tagged with source_secondary_id

completion_gate — current gate text (amended if a patch record has updated it)

completion_gate_history[] — previous gate versions with amendment source and timestamp

prerequisite_objective_ids — array; used for dependency-gated sequencing

status — computed: locked | available | active | complete



9.3 Deduplication rule

If a secondary objective would duplicate an objective already loaded by the primary (matched on objective_id), the secondary version is not loaded. Modifying secondaries expand existing objectives via patch records — they do not create duplicate objectives.






10. Developer acceptance criteria



Criterion

Requirement

Project creation — type selection

Exactly one primary type can be selected. Zero to N secondary types can be selected. Saving with no primary is blocked.

Design tension detection

If a selected combination matches a known tension pair, a named warning is surfaced before the project is saved. The steward must actively acknowledge before proceeding. Acknowledgement is recorded with timestamp.

Tension reminder at resolution tier

Each tension is surfaced again as a contextual reminder at the specific Plan tier where it resolves (per the tension register). This is a reminder, not a repeated acknowledgement gate.

Objective loading

On project creation: all universal objectives + primary-type objectives + delta objectives from each secondary, deduplicated. No objective appears twice.

Secondary classification

Modifying secondaries must update scope or checklist content of affected primary objectives via patch records. Classification is evaluated per primary–secondary pair, not per secondary type alone.

Sparse tiers

A project type may produce zero type-specific objectives in one or more tiers. Sparse tiers are valid. The UI renders them with an explanatory note, not as errors. Universal objectives ensure no tier is ever completely empty.

Mid-project secondary addition

Runs delta computation. Only new objectives inserted. Completed work is not re-gated silently. If patch records reopen a completed objective, a named modal alert is shown listing each affected objective with a direct link.

Reopening alert

The alert text must name the secondary being added, the number of objectives affected, and link directly to each. The steward actively acknowledges before continuing. The alert is never a badge or silent state change.

Completion gate amendments

If a patch record carries a completion_gate_amendment, the target objective's completion gate is updated to the amended text. The previous gate text is preserved in version history.

Secondary removal

Removing a secondary hides its delta objectives only if none have been started. If any delta objective is active or complete, removal is blocked with an explanation naming the blocking objectives.

Project type record

All changes — initial selection, secondary addition, removal, tension acknowledgements, reopening acknowledgements — are persisted as append-only versioned entries with timestamp and actor. Never overwritten.
