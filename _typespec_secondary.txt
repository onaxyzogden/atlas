1. Purpose & architectural context
OLOS organises land stewardship work around a universal 7-tier Plan stage. The tier structure is fixed — it reflects the epistemic dependency chain of regenerative land design. What varies by project is which objectives are loaded within each tier.
The Project Type & Secondary Layer system is the configuration mechanism that determines which objectives populate the universal tier structure for a given project. It operates on three layers: a universal objective set present for every project regardless of type; a primary-type objective set that anchors the project's core logic; and secondary objective sets that expand the project via additive or modifying behaviour.

[TABLE]
Architectural principle  The tier structure encodes how land design must proceed. Project type encodes what kind of land project is being designed. These are separate concerns and must remain so in the data model.
[/TABLE]

3. Primary / secondary layer architecture
3.1 Primary type
The primary type defines the core purpose of the land project. It anchors the objective structure across all tiers and sets the dominant design logic. Exactly one primary type is permitted per project.
3.2 Secondary types
Secondary types layer additional objectives and, where applicable, expanded scope onto the primary structure. Zero or more secondaries may be selected. There is no enforced upper limit — the design tension detection system (Section 6) provides practical guidance on combinations requiring careful resolution.
3.3 Project recipe examples

[TABLE]
Project | Primary | Secondaries | System behaviour
Ecovillage with food forest and retreats | Intentional Community / Ecovillage | Orchard / Food Forest + Agritourism / Retreat | Governance + privacy logic anchors all tiers. Food Forest is additive on Ecovillage (Tier 4–5 objectives). Agritourism is additive and triggers design tension #5 warning.
Family homestead with egg sales | Homestead / Family Land | Silvopasture / Livestock | Homestead logic anchors all tiers. Silvopasture is modifying: expands water strategy scope (Tier 3) and Tier 5 animal integration objectives.
Regenerative farm with workshops and lodging | Regenerative Farm / Multi-Enterprise | Education / Demonstration + Agritourism / Retreat | Farm anchors Tiers 0–4. Both secondaries are additive. No design tension flagged.
Market garden with conservation margins | Market Garden / Intensive Crop Production | Conservation / Rewilding | Market garden is primary. Conservation triggers design tension #2: intervention philosophy conflict. OLOS surfaces spatial separation requirement.
[/TABLE]

4. Universal objective set
Sixteen objectives are universal — present in every OLOS project regardless of primary or secondary type selection. They cannot be removed by project type configuration. They represent the irreducible minimum of what every regenerative land project must decide before Act begins.

[TABLE]
Architectural principle  Universal objectives ensure no project can reach Act without having resolved the foundational design questions that all land work depends on. They are not defaults that can be toggled off — they are structural requirements.
[/TABLE]


[TABLE]
Tier | Tier Name | Universal Objective | Rationale
0 | Project Foundation | Define vision, goals & stewardship capacity | Anchors every project to human intention and resource reality. No land design proceeds without a stated purpose.
0 | Project Foundation | Establish site boundaries & legal constraints | Every project operates within a spatial and legal frame. Required before any survey or design work.
0 | Project Foundation | Map stakeholders & community relationships | Every project has people around it: neighbours, family, shared boundaries, local authority, Indigenous title. Skipping this produces expensive surprises in Tier 4 and Tier 6. Parallelisable with boundaries objective.
1 | Land Reading | Survey terrain & topography | Slope, aspect, and elevation affect every land decision without exception.
1 | Land Reading | Survey climate & sectors | Sun, wind, frost, and rainfall patterns apply to every project type.
1 | Land Reading | Survey existing ecology & habitat | Baseline ecology is required even for high-intervention projects. Parallelisable with all other Tier 1 surveys.
1 | Land Reading | Survey existing infrastructure & access | Every project has an existing access and infrastructure context that constrains all design.
2 | Systems Reading | Survey water movement & hydrology | Water is foundational to every land system. Requires terrain survey complete.
2 | Systems Reading | Survey soil conditions & subsurface | Soil health and subsurface constraints affect every land use without exception.
3 | Foundation Decisions | Confirm project direction & feasibility | Decision checkpoint: validates or revises the Tier 0 vision against all survey findings before design begins.
3 | Foundation Decisions | Define water strategy | Water decisions gate all downstream design. Universal across all project types; completion output absorbs type-specific variation.
3 | Foundation Decisions | Define spatial framework & zones | Every project requires a spatial allocation decision before Tier 4 design begins. Objective title and checklist adapt by type (e.g. 'Habitat & Intervention Zone Mapping' for Conservation).
4 | Design | Design access & circulation | Every project moves people, materials, and/or animals. No exceptions.
4 | Design | Design water harvesting & storage infrastructure | Universal; valid completion outputs include 'mains supply confirmed — harvesting not required' for projects without on-site collection needs.
4 | Design | Design soil improvement strategy | Every project manages fertility, even minimally. Low-intervention projects produce a 'minimal intervention confirmed' output.
5 | Integration | Design monitoring & observation system | Every project needs a defined feedback mechanism connecting Act completion back to Observe. The bridge between implementation and learning.
6 | Phasing & Resourcing | Define Phase 1 implementation plan | Every project requires a scoped first action phase before Act begins.
6 | Phasing & Resourcing | Define resource & capacity plan | Every project operates within labour, budget, and skill constraints that must be explicitly recorded.
6 | Phasing & Resourcing | Define risk & contingency register | Every project requires a named first-cycle risk register before Act begins. Without it, the Phase 1 plan is incomplete.
[/TABLE]

Tier 0 parallelism note
Within Tier 0, the Establish boundaries objective and the Map stakeholders objective are parallelisable. Both require the vision objective to be complete first. Neither can proceed without it. Both must be complete before Tier 1 unlocks.
Completion output flexibility
Universal objectives accommodate project-type variation through their completion outputs, not through separate objective instances. Examples: Zone Allocation accepts 'Habitat & Intervention Zone Mapping' as a valid output for Conservation projects. Water Harvesting Infrastructure accepts 'Mains supply confirmed — harvesting not required' as a valid output. Soil Improvement accepts 'Minimal intervention confirmed' for Conservation and Rewilding projects. The objective is universal; the checklist depth and valid outputs adapt by type.
5. Secondary type classification
5.1 Additive vs. modifying
Not all secondaries behave the same way. The system distinguishes two secondary behaviour types. This distinction determines whether a secondary appends new objectives only, or also reaches into existing primary objectives and expands their scope via patch records (see Section 7).

[TABLE]
Secondary type | Behaviour | Examples
Additive | Adds new objectives in Tiers 4–6 without changing existing primary objectives | Education, Agritourism, Wellness (most combinations)
Modifying | Expands the depth or scope of existing primary objectives in addition to adding new ones. Classification is a property of the secondary–primary relationship, not of the secondary type alone — the same secondary may be additive on one primary and modifying on another. | Silvopasture on Homestead (modifying); Silvopasture on Ecovillage (additive)
[/TABLE]


[TABLE]
Critical implementation note  Secondary classification is a property of the secondary–primary relationship, not of the secondary type alone. The same secondary may be modifying on one primary and additive on another. The relationship matrix in Section 5.3 is the canonical reference.
[/TABLE]

5.2 Classification key

[TABLE]
Symbol | Classification | Meaning
M | Modifying | Secondary expands scope of existing primary objectives via patch records, in addition to adding new objectives
A | Additive | Secondary appends new objectives only. No existing objective is modified.
X | Design Tension | Combination is possible but carries a known spatial or philosophical conflict. Warning surfaced at project creation and at the resolution tier.
— | Same type | Secondary and primary are the same project type. Not applicable.
[/TABLE]

Matrix patterns
Modifying secondaries cluster around Regenerative Farm, Market Garden, and Off-Grid — the use-defined types with the deepest operational infrastructure.
Education is universally additive. It never modifies a primary — it always appends curriculum, demonstration, and signage objectives on top.
Silvopasture is the most tensioned secondary — three X combinations. Animals create genuine conflict with intensive crops, conservation ecology, and low-stimulation sanctuaries.
Agritourism has one tension (Ecovillage) and is additive on all other primaries. Visitor experience layers sit on top of any land use without restructuring it.
6. Design tension detection
Ten project type combinations carry known spatial or philosophical conflicts. OLOS surfaces these as named design tension warnings at two points: at project creation when the combination is first selected, and as a contextual reminder at the specific Plan tier where the tension must be resolved.
Tensions are advisory — they do not block project creation or secondary addition. The steward must actively acknowledge each warning before proceeding. Acknowledgements are recorded in the project type version history.

[TABLE]
# | Type A | Type B | Tension description | Resolution tier
1 | Wellness / Healing Sanctuary | Agritourism / Retreat | Quiet vs. visitor traffic. Sanctuary design requires low-stimulation, privacy-graded zones. High-traffic guest programs conflict unless spatially separated. Advisory, not blocking — the steward may be designing a quiet retreat, not a high-traffic operation. Resolution anchored to Tier 3 Zone Allocation. | Tier 3 — Zone Allocation
2 | Conservation / Rewilding | Market Garden / Intensive Crop Production | Intervention philosophy conflict. High-input annual production contradicts minimal-intervention habitat logic. Requires hard spatial boundaries between production and restoration zones. | Tier 3 — Zone Allocation
3 | Conservation / Rewilding | Silvopasture / Livestock | Animal impact vs. habitat recovery. Grazing pressure, compaction, and browse damage threaten sensitive restoration areas. Requires fencing, exclusion zones, and corridor design before Act. | Tier 3 — Zone Allocation, Tier 5
4 | Off-Grid Resilience | Education / Demonstration Land | Access conflict. Remote settlement design minimises and controls site access. Education requires regular, safe, predictable public access. Driveway, parking, and security design must reconcile both. | Tier 4 — Access & Circulation
5 | Intentional Community / Ecovillage | Agritourism / Retreat | Governance and privacy conflict. Resident consent, community agreements, and private living zones may directly conflict with open visitor access. Requires explicit visitor policy before Zone Allocation. | Tier 3 — Zone Allocation
6 | Silvopasture / Livestock | Market Garden / Intensive Crop Production | Contamination and damage risk. Animal movement near intensive crop beds creates pathogen, compaction, and browse risk. Requires strict spatial and temporal separation. | Tier 4 — Access & Circulation, Design
7 | Silvopasture / Livestock | Wellness / Healing Sanctuary | Sensory conflict. Animal noise, odour, and operational activity are incompatible with low-stimulation sanctuary zones unless sufficient distance and screening is designed in. | Tier 3 — Zone Allocation
8 | Off-Grid Resilience | Agritourism / Retreat | Access conflict. Remote settlement deliberately limits visitor access for security and privacy. Agritourism requires regular, predictable, open public access. Driveway, security, and visitor management design must reconcile both. | Tier 4 — Access & Circulation
9 | Residential / Live-In Stewardship | Agritourism / Retreat | Private residence vs. visitor access conflict. The steward's home is on the same land guests are visiting. Private living zones, family security, and household routines must be explicitly separated from visitor circulation before Zone Allocation. | Tier 3 — Zone Allocation
10 | Residential / Live-In Stewardship | Wellness / Healing Sanctuary | Private domestic life vs. therapeutic sanctuary zones. Household noise, activity, and domestic infrastructure conflict with the low-stimulation, privacy-graded environment required for a healing sanctuary. Requires hard spatial separation and acoustic buffering. | Tier 3 — Zone Allocation
[/TABLE]

6.1 Tension warning behaviour
Warning text names the specific conflict in plain language — not a generic alert.
Steward must actively acknowledge ('Understood — continue') before the project is saved or the secondary is added.
Acknowledgement is recorded: tension ID, timestamp, actor.
At the resolution tier, the tension is surfaced again as a contextual reminder in the objective detail panel. This is a reminder only — no second acknowledgement gate.
All 8 tensions are unconditional. The system surfaces them regardless of the steward's intended implementation approach. Resolution is the steward's responsibility in Zone Allocation or the relevant design tier.
7. Modifying secondary mechanism
7.1 The patching model
When a modifying secondary is applied to a project, it expands existing primary objectives by injecting additional checklist items into them. This is the patch record model. It keeps the objective unified — the steward works one checklist, not two — while ensuring the expanded scope is explicitly represented and traceable to its source.
7.2 Patch record structure
Each modifying secondary carries one or more patch records. Each patch record targets a single objective and defines the injection precisely.

[TABLE]
Field | Purpose
target_objective_id | References the existing objective that receives the patch
inject_after_item_id | Placement anchor within the target checklist. Null = append after last native item
items[] | The checklist items being injected. Each item carries: id, text, is_methodology boolean, feeds_into string
source_secondary_id | Tags each injected item to its originating secondary. Required for removal tracking and UI labelling
scope_note | Plain-language description of what the secondary adds to this objective. Displayed in the objective detail panel as: 'Scope expanded by: [Secondary name]'
completion_gate_amendment | Optional. Updated completion gate text for when the secondary changes what 'done' means for the objective. Example: water strategy completion gate expands from 'water strategy approved' to 'water strategy approved including stock water system' when Silvopasture is added
[/TABLE]

7.3 Conflict resolution — multiple secondaries patching the same objective
When two or more secondaries both carry patch records targeting the same objective, their injections are applied in the order the secondaries were added to the project. Injected items from each secondary append after the last native item (or after the previous secondary's injections if multiple). Each injected item retains its source_secondary_id tag regardless of ordering.
7.4 Reopening completed objectives
If a secondary is added after an objective has been marked complete, and that secondary's patch records inject items into that objective, the objective reverts to active status. Completed checklist items remain checked. Only the newly injected items appear as outstanding.

[TABLE]
UX requirement  This reopening must never be silent. A named modal alert is required — never a badge or a silent state change. The alert must name the secondary being added, list each affected objective, state how many new checklist items have been added, and provide a direct link to each affected objective. The steward actively acknowledges before continuing.
[/TABLE]

7.5 Completion gate amendments
Some modifying secondaries change not just what work is required, but what 'done' means for the objective. When a patch record carries a completion_gate_amendment field, the target objective's completion gate text is updated to the amended version. The previous gate text is preserved in the objective's version history. Both texts are shown in the detail panel: current gate (active) and previous gate (greyed, with amendment source noted).
7.6 Scope note display
The objective detail panel must display a scope note for any objective that has been expanded by one or more modifying secondaries. Format: 'Scope expanded by: [Secondary name 1], [Secondary name 2]'. This is shown below the objective description and above the checklist. It links directly to the secondary's project recipe entry.
8. Mid-project secondary addition
8.1 Design rationale
Land projects reveal themselves over time. Blocking secondary addition at project creation fights the actual rhythm of land stewardship. OLOS must accommodate evolving project scope without requiring a project restart.
8.2 Delta computation protocol

[TABLE]
Step | Behaviour
1. Audit completed work | Identify all objectives already complete or active. These are never re-gated.
2. Compute delta objectives | Identify objectives the new secondary introduces that do not already exist in the project.
3. Insert delta at correct tier | New objectives are placed in the appropriate tier per the universal tier structure. No new tiers are created.
4. Apply patch records | If the secondary is modifying on this primary, apply all patch records to their target objectives. If any target is complete, trigger the reopening alert.
5. Flag Observe gaps | If the secondary requires Observe data not yet collected, surface a named gap notification. Never fail silently.
6. Flag design tensions | If the secondary creates a tension with the existing primary or any active secondary, surface the tension warning immediately.
[/TABLE]

8.3 Secondary removal
Removing a secondary mid-project is permitted only if none of its delta objectives are active or complete. If any delta objective has been started or finished, removal is blocked. The UI must name which objective(s) are preventing removal and offer the option to mark them as Deferred instead.
8.4 Version history
Every change to the project type configuration — initial selection, secondary addition, secondary removal, tension acknowledgement, reopening acknowledgement — is recorded as an append-only versioned entry with action type, timestamp, and actor. This record is never overwritten.
9. Data model requirements
9.1 Project type record
project_id — foreign key to project
primary_type_id — reference to project type catalogue (exactly one)
secondary_type_ids — array of references to project type catalogue (zero or more)
created_at — timestamp of initial selection
version_history — append-only array of change events (see 8.4)
tension_acknowledgements — array of { tension_id, acknowledged_at, acknowledged_by }
reopening_acknowledgements — array of { secondary_id, affected_objective_ids[], acknowledged_at, acknowledged_by }
9.2 Objective record (relevant fields)
objective_id
tier — integer 0–6
source — enum: universal | primary | secondary
source_type_id — null if universal; references project type catalogue if primary or secondary
secondary_class — null if not a secondary objective; enum: additive | modifying
patch_records[] — array of patch records this objective carries when acting as a modifying secondary source
injected_items[] — checklist items injected from secondary patch records, each tagged with source_secondary_id
completion_gate — current gate text (amended if a patch record has updated it)
completion_gate_history[] — previous gate versions with amendment source and timestamp
prerequisite_objective_ids — array; used for dependency-gated sequencing
status — computed: locked | available | active | complete
9.3 Deduplication rule
If a secondary objective would duplicate an objective already loaded by the primary (matched on objective_id), the secondary version is not loaded. Modifying secondaries expand existing objectives via patch records — they do not create duplicate objectives.
10. Developer acceptance criteria

[TABLE]
Criterion | Requirement
Project creation — type selection | Exactly one primary type can be selected. Zero to N secondary types can be selected. Saving with no primary is blocked.
Design tension detection | If a selected combination matches a known tension pair, a named warning is surfaced before the project is saved. The steward must actively acknowledge before proceeding. Acknowledgement is recorded with timestamp.
Tension reminder at resolution tier | Each tension is surfaced again as a contextual reminder at the specific Plan tier where it resolves (per the tension register). This is a reminder, not a repeated acknowledgement gate.
Objective loading | On project creation: all universal objectives + primary-type objectives + delta objectives from each secondary, deduplicated. No objective appears twice.
Secondary classification | Modifying secondaries must update scope or checklist content of affected primary objectives via patch records. Classification is evaluated per primary–secondary pair, not per secondary type alone.
Sparse tiers | A project type may produce zero type-specific objectives in one or more tiers. Sparse tiers are valid. The UI renders them with an explanatory note, not as errors. Universal objectives ensure no tier is ever completely empty.
Mid-project secondary addition | Runs delta computation. Only new objectives inserted. Completed work is not re-gated silently. If patch records reopen a completed objective, a named modal alert is shown listing each affected objective with a direct link.
Reopening alert | The alert text must name the secondary being added, the number of objectives affected, and link directly to each. The steward actively acknowledges before continuing. The alert is never a badge or silent state change.
Completion gate amendments | If a patch record carries a completion_gate_amendment, the target objective's completion gate is updated to the amended text. The previous gate text is preserved in version history.
Secondary removal | Removing a secondary hides its delta objectives only if none have been started. If any delta objective is active or complete, removal is blocked with an explanation naming the blocking objectives.
Project type record | All changes — initial selection, secondary addition, removal, tension acknowledgements, reopening acknowledgements — are persisted as append-only versioned entries with timestamp and actor. Never overwritten.
[/TABLE]
