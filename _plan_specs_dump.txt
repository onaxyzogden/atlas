OLOS Plan Stage Developer Specification

OLOS Plan Stage Developer Specification

Domain → Objective → Decision Checklist → Map Overlays → Plan Output → Act Handoff



Purpose of this document

This specification defines the Plan stage of OLOS as a developer-ready workflow. It keeps Plan focused on translating observed evidence into design decisions, sequencing, suitability judgments, scoped work packages, and approval-ready handoffs. It prevents Observe-stage data collection and Act-stage execution from leaking into Plan.

Core principle:

Observe records reality. Plan makes decisions. Act coordinates execution.

Prepared for developer implementation of the task-focused OLOS map view component.

1. Executive Summary

OLOS is organized around three main stages: Observe, Plan, and Act. Each stage uses the same land-based Domains, but the user’s relationship to each Domain changes by stage. Plan is where observed reality becomes intentional design. It is not for gathering raw baseline evidence and it is not for assigning field execution. Plan is for deciding what should happen, where it should happen, why it should happen there, in what sequence, under what conditions, and what must be handed to Act for coordinated completion.

Developer takeaway

When the user selects Plan + Domain + Objective, the map view should load only the overlays, decision checklist, supporting Observe inputs, alternatives, constraints, and approval fields required for that planning task. The user should not see every property layer at once.

3. Plan Stage Boundary Rules

Plan definition

Plan is the decision-making and design stage of OLOS. Its purpose is to convert Observe-stage evidence into selected strategies, spatial layouts, system designs, phasing decisions, dependencies, prerequisites, constraints, and approval-ready Act handoff packages.

Allowed Plan verbs

Decide

Design

Compare

Select

Reject

Prioritize

Sequence

Phase

Estimate

Approve

Define

Specify

Constrain

Resolve

Defer

Avoid in Plan

Collect raw baseline observations without reference to a decision

Treat unknowns as solved

Assign workers to field execution

Mark tasks complete

Perform construction

Manage daily operations

Verify physical completion

Replace legal, engineering, ecological, or professional review when required

Plan outputs instead of premature execution

When Plan reaches a decision, it should produce a status such as Approved for Act, Needs More Observation, Needs Professional Review, Redesign Required, Deferred, or Rejected. Act should only receive approved or conditionally approved handoff packages.

4. Map View Behavior

The Plan map view should be a decision workspace. It should show the selected Domain, the specific Objective, relevant Observe inputs, possible design options, the overlays needed to compare them, and a side-panel checklist that moves the user toward an approval-ready plan.

5. Universal Overlays

Overlay

Purpose in Plan

Zones

Frequency of human attention and maintenance intensity by area. Used to place systems according to how often people must interact with them.

Sectors

Incoming external forces such as sun, wind, storms, wildfire path, wildlife corridors, noise, road exposure, smoke, odour, views, and neighboring influence.

Contours / Landform

Slope, aspect, elevation, high points, low points, ridgelines, valleys, depressions, and erosion paths.

Water Flow

Runoff, water entry/exit points, wet areas, dry areas, drainage, ponds, wells, springs, roof catchment, and flood-prone areas.

Soil Conditions

Soil type, compaction, infiltration, erosion, contamination, organic matter, test zones, and visible soil health indicators.

Ecology / Habitat

Vegetation communities, wildlife corridors, invasive species, sensitive habitat, pollinator areas, wetlands, and restoration areas.

Access / Movement

Roads, paths, gates, parking, emergency access, equipment routes, animal lanes, delivery points, and harvest routes.

Infrastructure / Utilities

Buildings, fences, greenhouses, barns, storage, utilities, water lines, power, compost areas, and other physical assets.

Resource Flows

Energy, water, compost, manure, mulch, feed, greywater, waste, materials, and storage flows.

Roles / Responsibility

Stewards, owners, workers, volunteers, decision rights, permissions, accountability zones, and reporting responsibilities.

Risk / Compliance

Zoning, setbacks, permits, easements, fire risk, flood risk, contamination, protected areas, safety hazards, and legal constraints.

Suitability

Suitability or constraint status for uses such as dwelling, growing, grazing, building, access, restoration, water storage, or retreat use.

Stewardship Intensity

Care frequency: daily, weekly, monthly, seasonal, annual, passive/self-regulating, or unknown.

Monitoring / Records

Observation points, photo points, seasonal logs, tests, sensor readings, previous records, evidence, and before/after documentation.

Timeline / Phasing

Existing condition, immediate constraints, Year 0 setup, Year 1 establishment, Year 3 maturity, full vision, and ongoing stewardship.

1. Vision & Project Intent

Converts the Stage Zero vision and Observe-stage feasibility evidence into a clear, constrained project direction.

Field

Developer Specification

Plan objective

Define the project direction and confirm which parts of the vision are feasible, conditional, deferred, or rejected.

Planning question

Given the observed conditions, what version of the intended land project should be planned?

Required Observe inputs

Stage Zero vision; project type; non-negotiables; disqualifying feature flags; land suitability observations; legal/risk flags; capacity baseline

Default map overlays

Suitability; Risk / Compliance; Timeline / Phasing; Economics & Capacity indicators; Monitoring / Records

Checklist items

Review stated vision and non-negotiables

Review all Observe-stage disqualifier flags

Classify vision elements as feasible, conditional, deferred, or rejected

Identify minimum viable project scope

Define success criteria for the first planning cycle

Record assumptions and unresolved questions

Approve a bounded planning direction

Decision inputs / evidence

Stage Zero answers; Observe handoff packets; feasibility notes; capacity evidence; legal/risk documents; stakeholder constraints

Completion criteria

A bounded, evidence-informed project direction is approved or sent back for more observation/review.

Act handoff output

Project Direction Brief: approved scope, deferred elements, rejected elements, assumptions, constraints, and first-cycle success criteria.

Allowed outputs

Approved for Act; Conditionally Approved; Needs More Observation; Needs Professional Review; Redesign Required; Deferred; Rejected

1. Context & architectural decisions



1.1 The three-stage cycle

OLOS organises all land stewardship work around three stages: Plan, Act, and Observe.



Stage

Role

Plan

Determines what needs to be known and done. The entry point for any new project. Sequences objectives by dependency, generates Act tasks, and gates downstream decisions behind genuine prerequisite completion.

Act

Executes all field work — both survey/observation tasks and physical implementation tasks. These feel identical in the UI. Completed Act tasks feed data into Observe.

Observe

A read-only intelligence layer. Surfaces synthesised data produced by Act tasks. The user never 'does' Observe — they read it. It grows richer as Act tasks complete.



Key architectural principle

Observe is not the entry point. Plan is. The cycle is: Plan (decide what to learn or do) → Act (execute it) → Observe (read what was learned). A new project always begins in Plan.



1.2 Objectives as the primary navigation unit

The previous navigation model used Domain as the entry point (Stage → Domain → Objective). This has been replaced.



The problem with domain-first navigation:

Objectives routinely cross domain boundaries. “Map water movement” requires Topography, Hydrology, Soil, and Climate data simultaneously. Forcing it into a single domain produced incomplete decisions.

Free domain selection allowed users to plan in the wrong order — e.g. designing planting systems before water strategy was resolved — producing expensive rework.

Domain was an organisational lens for data, not a unit of work. Objectives are the actual unit of work.



The new model:

Stage → Objective (domain overlays load automatically based on what the objective requires)

The user never selects a domain. They select what they are trying to understand or decide.

Each objective carries its required overlays, prerequisite objectives, parallelisable peers, and completion gate as first-class data.



1.3 Dependency-gated sequencing

Regenerative land design has hard dependencies that, if violated, produce rework. The Plan stage enforces these as a tier-based dependency graph. A user cannot access Tier N until Tier N-1 is complete (with parallelism within tiers where applicable).



The dependency graph is not paternalistic — it reflects real ecological and design logic:

Water movement cannot be correctly read without terrain data.

Zone allocation requires terrain, water, sectors, and access observations.

Planting system design without resolved water strategy produces redesign.



Locked objectives are informative, not punitive. Clicking a locked tier reveals a plain-language explanation of which prerequisite must complete first.

2.8 The Plan–Act–Observe spiral & two-mode gate logic



2.8.1 The cycle is a spiral, not a sequence

Plan → Act → Observe is not a one-time linear sequence. It is a recurring spiral. Act execution changes the land. Observe surfaces what changed. Plan must then revisit prior decisions in light of new evidence and adapt.



This has a direct consequence for the tier structure: the gate logic that governs Tier access must behave differently depending on whether the user is in their initial pass or a subsequent cycle.



2.8.2 Two gate modes



Mode

When it applies

Gate behaviour

Initial completion

First time through the Plan stage on a new project

Hard gates. A tier is inaccessible until all prerequisite objectives are genuinely complete. The user cannot skip or bypass.

Cyclical review

Any subsequent Plan cycle after Act has executed and Observe has updated

Soft gates. Completed tiers become review checkpoints rather than locked barriers. The user can open any previously completed tier to reassess and revise decisions in light of new Observe data.



Why this matters

In year 3 of a project, a steward should not be forced to re-complete a Terrain Survey before they can revise their Water Strategy. The terrain is known. What has changed is what Observe has surfaced since the last cycle. The gate should ask: has anything changed that affects this tier? — not: have you completed all prerequisites from scratch?

In cyclical review mode, the tier spine communicates the shift:

Blue dot

Lock icon + hard gate

No locked tiers in review mode — all previously completed tiers are accessible

Tier not yet reached

Locked as normal

Locked as normal — tiers the user has never completed remain hard-gated



When a user opens a tier in review mode, the objective cards show their last-completed state with a “Review” callout noting the Observe data that has changed since the last cycle. The user can update decisions and re-approve objectives without re-completing the full checklist from scratch.



2.8.4 Cyclical review UI — entry and confirmation screens

The two-mode gate logic described in sections 2.8.1–2.8.3 requires two distinct UI screens to make the review flow legible to stewards.



Screen 1 — Review entry state

When Observe data has updated enough to trigger a Plan revision signal, and the steward enters a tier in cyclical review mode, a modal or banner appears before the tier’s objectives are shown. It explains in plain language what changed and what may be affected.



Element

Specification

What Observe detected

One or two sentences stating the specific data change. Example: “Soil pH at Zone 2 has dropped from 6.4 to 5.8 since your last review.” Not a generic alert — always specific to the actual data.

A compact list of the affected objectives with their tier and a one-line reason each is flagged. Each item is tappable.

A compact one-line summary in the detail panel below the objective title: ‘Confirmed on [date]: [decision summary]’ or ‘Updated on [date]: [what changed]’.

Review mode feels different from first-cycle mode

In cyclical review, the steward is not completing things for the first time. They are assessing whether prior decisions still hold. The UI must communicate this difference — amber review badges, confirmation language, and the Observe data context panel all signal that this is a revision session, not initial planning.



2.8.4 Tier 3 naming rationale

Tier 3 is named “Foundation Decisions” rather than “First Design Decisions” precisely because of the spiral. The zone framework and water strategy are the foundational decisions in every cycle — they are the ones everything else depends on whether you are making them for the first time or revising them in year 5. The word “first” would become incorrect and confusing on every cycle after the initial one.

5. Stage boundary rules & covenant constraints



5.1 What belongs in Plan

Plan objectives are decision-making surfaces. Allowed verbs: decide, design, compare, select, reject, prioritise, sequence, phase, estimate, approve, define, specify, constrain, resolve, defer.



Plan must not:

Collect raw baseline observations without reference to a decision

Assign workers to field execution

Mark Act tasks complete

Perform construction or physical implementation

Replace legal, engineering, ecological, or professional review when required



5.2 Plan outputs

When a Plan objective reaches a decision, it produces one of the following statuses:

Approved for Act

Conditionally Approved (with named conditions)

Needs More Observation (flags a gap in Observe data)

Needs Professional Review

Redesign Required

Deferred

Rejected



Act should only receive Approved or Conditionally Approved handoff packages.



5.3 Covenant boundary — Plan vs Economics

Covenant constraint

Tier 6 (Phasing & Resourcing) covers project cost and budget tracking only. Capital formation, financing, advance-purchase agreements, investor or equity structures, and yield-as-return framing are explicitly excluded from Plan and fall under Sub-project C (Economics), which is governed by Scholar Council review. No riba or gharar framing in any Plan surface.




6. Developer acceptance criteria



6.1 Navigation

A new project enters at Tier 0. No other tier is accessible until Tier 0 is complete.

Clicking a locked tier reveals an inline callout explaining the prerequisite. Clicking again dismisses it. The tier row is never blocked or inert.

Clicking an unlocked tier switches the centre panel to that tier’s objectives without page reload.

Selecting an objective populates the detail panel without navigating away from the Command Center.



6.2 Next up affordance

The Next Up card always reflects the current project state — it is not static.

If an active objective exists, Next Up shows “Continue where you left off” with correct step count.

If no active objective but an available one exists, Next Up shows “Start here.”

If all objectives in the current tier are complete, Next Up does not show (the tier is complete; user should advance to the next tier).



6.3 Progress bar

Progress percentage and objective count update immediately on checklist item interaction — no page reload.

Progress reflects completed objectives across all tiers, not just the active tier.

Stage pills correctly indicate current stage and navigate to other stage command centers on click.



6.4 Detail panel

Map activation strip shows the correct overlay count and first two overlay names for the selected objective.

Checklist items persist state within the session. Checked items show strikethrough.

Done count in the checklist label updates live.

Details toggle is collapsed by default. State persists within the session (if user expanded it, it stays expanded when they return to the same objective).

Status is computed from checklist state and prerequisite completion — it is never manually set.

Completing the final checklist item of an objective does not automatically mark it complete — the completion gate must also be met.

Uses the same organic circle motif from the active tier — the newly unlocked tier number rendered large in the organic circle, centre-aligned in the banner.



The goal is felt difference, not visual noise

Every visual language change should pass this test: does it make the product feel more like it was designed for land? If the answer is no, or if the change adds visual noise rather than meaning, do not implement it. Subtle and intentional outperforms decorative every time.

1. System overview



1.1 What OLOS is

OLOS is the full operating system for regenerative land development. Its success is measured by whether it can independently guide a project through observation, design, phased implementation, daily management, ecological monitoring, and adaptive stewardship at the complexity level of Apricot Lane Farms — without requiring external project management systems or personnel to hold the work together.



1.2 The three-stage cycle

All work in OLOS moves through a recurring spiral of three stages:



Stage

Role in the system

Plan

Determines what needs to be known and done. Sequences objectives by dependency. The entry point for every new project and every new cycle. Generates Act tasks.

Act

Executes all field work — survey tasks and implementation tasks are treated identically in the UI. Collects proof. Feeds verified data into Observe. The primary mobile surface.

Observe

Read-only intelligence layer. Surfaces synthesised data produced by Act. Grows richer across cycles. Feeds back into Plan when conditions change enough to warrant revision.



The cycle is a spiral, not a sequence

Plan → Act → Observe repeats. Each cycle produces ground truth that informs the next Plan revision. The system is designed for this from the data model up — every data point carries a cycle ID, every Plan objective has a review mode for subsequent cycles, and the Observe Plan revision banner closes the loop automatically.

3. Ratified architectural decisions



The following decisions were made during spec development and are binding for implementation. They resolve questions that would otherwise be re-litigated during build.



Navigation & information architecture

Tier unlock transitions display a full-width celebration banner in the centre panel with a ‘What just became possible’ section. The lock icon animates away on unlock. This is the primary moment of felt progress.

Locked tiers open a popover (not a modal) with lock reason, prerequisite progress indicator, and a subtle ‘open anyway’ bypass link. An amber out-of-sequence banner appears if the user proceeds.

Domain is a tag on an objective, not a navigation gate. The user never selects a domain. Overlays load automatically based on what the objective requires.

Stage navigation is Stage → Objective, not Stage → Domain → Objective.

The 7-stage lifecycle is deprecated. The 3-stage cycle (Plan / Act / Observe) is the forward IA.

Tier 3 is named ‘Foundation Decisions’, not ‘First Design Decisions’. The name is cycle-neutral by design.



Plan stage

The detail panel is organised into four explicit sections: OBJECTIVE, MAP ACTIVATION, YOUR DECISIONS (with feeds-into tags), and REFERENCE (collapsed by default).

Each decision item carries a ‘feeds into →’ tag showing its downstream consequence. Tags are authored per item via the feeds_into field on ChecklistItem. When checked, the tag briefly highlights green.

Cyclical review entry shows a modal explaining what Observe detected and which objectives are flagged. Confirmation shows a ‘Decision confirmed’ or ‘Decision updated’ badge and a summary line.

Tier gate logic has two modes: initial completion (hard gates) and cyclical review (soft gates — previously completed tiers become review checkpoints).

Objective status is computed, not manually set. Resolution order: complete → active → available → locked.

Plan outputs are one of: Approved for Act, Conditionally Approved, Needs More Observation, Needs Professional Review, Redesign Required, Deferred, Rejected.



Act stage

The Launch Act Command Center transition collapses the left sidebar to objective-relevant tools only, pre-activates overlays on the map, and shows a floating bottom-right card with objective name, decision progress, and a Back to Plan button.

Survey tasks and implementation tasks are treated identically in the UI. The distinction matters for data routing only.

Proof schemas are task-type specific. A universal catch-all schema is explicitly rejected.

Checklist steps in survey tasks are proof methodology (is_methodology: true) and carry evidential weight. In implementation tasks they are operational guidance.

Act is mobile-first. The three-panel Plan layout must not be applied to Act.

View A (objective execution) must ship with the Act map view. They are a single unit.

Objective overlays are the default active set on map open. The full project overlay library remains accessible via the layers control.

Reality diverges is a first-class action available on every task. It is not an error state.

Divergence evidence routes to Observe unconditionally and permanently, regardless of Plan revision status.



Observe stage

Observe is read-only. It is populated exclusively by verified Act tasks and divergence captures.

No health scores, traffic-light ratings, or aggregated wellness indices. Observe shows evidence; it does not evaluate it.

Observe data is never edited or deleted from within the Observe surface. Corrections require a new verified Act task that supersedes the prior data point.

Superseded data points remain visible and accessible. They are never hidden.

The Plan revision banner triggers are tiered: Critical (undismissable), High (dismissible with reminder), Informational (summary row only).

Presentation mode excludes all internal planning state. It surfaces verified evidence only.



Project Home

The Next Up card surfaces exactly one item. Showing multiple items hands the prioritisation decision back to the steward, defeating the card’s purpose.

Urgency score is never displayed as a number.

Contractor access is enforced at the data layer. It is not a UI-only restriction.

Contractor access expires under three conditions: all tasks verified, manual revocation, or 90-day time fallback.



Notifications

Interrupt only when the cost of delay is concrete.

Critical-tier notifications cannot be opted out of. Only delivery address can be changed.

Informational notifications never trigger email or push.

Safety divergence and protected species discovery notifications are delivered to all project team members regardless of role — the safety override crosses all role boundaries.

Escalation updates the existing notification in-place rather than creating a new entry.

The 14-day inactivity notification defaults to email on for primary stewards. It is the one Informational event with email default-on.

The following sequence minimises blocked work and surfaces value to real users at each milestone.

Phase 1 is the make-or-break phase

If a steward can create a project by walking their boundary, name their land, and see their first four survey objectives ready to start — all in under 10 minutes — the system has earned trust. Every subsequent phase builds on that trust. If Phase 1 feels slow, bureaucratic, or spatially disconnected, later phases will not recover it.




6. Open questions for implementation team



The following questions were intentionally left to the implementation team. They are infrastructure or product decisions that fall outside UX spec scope.



Question

Context

What is the offline sync conflict resolution strategy?

Act spec requires local-first data model. When two team members modify the same task offline simultaneously, which write wins? Last-write-wins is simplest but may discard proof items.

What map tile provider is used?

Act map view and wizard boundary drawing depend on a tile provider. Provider affects offline caching strategy, usage costs, and available satellite imagery quality.

What is the GPS accuracy threshold for walk-the-boundary mode?

Act spec notes an accuracy threshold for GPS walk traces but defers the specific value. Recommend ±10m for boundaries, ±5m for survey traces.

How are cycle IDs generated and scoped?

Observe spec defines cycle IDs as immutable once assigned. The triggering condition for a new cycle (steward initiates a new Plan stage after Act completion?) needs engineering definition.

What is the maximum project site area supported?

No upper bound is defined. Very large stations (50,000+ ha) may require tile caching strategies different from small holdings.

How are measurement units handled globally?

Act proof schema uses measurements with units. The system should respect locale defaults (metric vs imperial) but allow per-project override.
