## Purpose
Owns the design-time temporal model of how a property is built out over
time: a Year 0 / Year 1 / Year 3 / Year 5 / Full Vision timeline slider,
user-defined phases with stage dependencies and milestones, a build
sequence map sequencing infrastructure, water, regeneration, and revenue
work, a temporary-vs-permanent element view with seasonal overlays, and
per-phase cost/labor/material aggregates. Higher-phase capabilities add
permit dependency warnings with readiness checklists, per-phase
completion tracking with notes, scenario-based phasing alternatives, and
four preset optimization modes ("Fastest feasible path", "Lowest cost
path", "Most regenerative path", "Investor presentation path") that
re-order phases against different objective functions.

## Known gotchas
- `phaseStore` persists under a Zustand `persist` middleware — any
  schema change to `BuildPhase` must bump the persist version or
  migrate, or stale browsers will hydrate half-shaped phases.
- The timeline currently assumes exactly four default phases
  (`Year 0-1`, `1-3`, `3-5`, `5+`). The spec's five timeline-slider
  modes (Year 0 / Year 1 / Year 3 / Year 5 / Full Vision) are a
  *view* on top of whatever custom phases the user defines — do not
  hard-code five phases; scrub modes are read-only projections.
- `computeAssessmentScores` and `computeOverallScore` are imported by
  `TimelinePanel` for the Complexity tab. When Section 4's scoring
  migration to `@ogden/shared/scoring` completes, migrate these
  imports too — do not let the timeline be the last consumer of the
  legacy web-side `lib/computeScores.ts`.
- Phase dependency violations in
  `utilityAnalysis.checkDependencyViolations` only cover utilities.
  Extending to structures/crops/paths requires a shared dependency-
  graph type (good candidate for the Zod schema upgrade).
- Path-mode preset ordering ("Fastest feasible", "Lowest cost", etc.)
  needs an explicit objective function per mode — do not ship a vague
  heuristic; each mode must be reproducible from current store state.
- `aggregatePhaseFeatures` in `timelineHelpers.ts` takes element lists
  by reference — memoize callers or the timeline will re-render on
  every unrelated store mutation.
