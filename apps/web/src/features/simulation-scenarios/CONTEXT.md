## Phase tags
P3, P4 (see `packages/shared/src/featureManifest.ts` for the authoritative item list)

## Purpose
Provide the what-if surface: run forward simulations and side-by-side scenario
comparisons for water flow/flood/drought, wind and shade growth, erosion and
grazing recovery, carrying capacity and yield, visitor/event load, and build
cost/revenue. Users create best-case/base/worst-case variants, compare alternate
layouts (A/B/C), run sensitivity analysis on key assumptions, and overlay
climate-shift projections. The section answers "what happens if…?" questions on
top of the design surfaces owned by Sections 5–15.

## Known gotchas
- This section intentionally overlaps with `features/scenarios/`. Three items
  are flagged `partial` because that surface already handles scenario-snapshot
  comparison and financial ranges; all remaining items are `planned` and need
  real simulation engines before UI can land.
- Forward simulations (water flow, wind, erosion, carrying capacity) are
  compute-heavy. Do not plan to run them in the browser — design server
  endpoints and cache results per scenario id.
- Climate-shift overlays (P4) are the one P4-tagged item; the section as a
  whole defaults to P3. Feature gating must read per-item phase, not the
  section header.
- Best/base/worst-case snapshots must reuse `ScenarioPanel`'s variant-config
  shape (`projectType`, `region`, `missionFocus`) rather than inventing a
  parallel schema.
- Sensitivity analysis implies re-running the financial engine N times with
  perturbed inputs — `computeMissionScore` is not pure yet; audit before
  wiring sweeps.
