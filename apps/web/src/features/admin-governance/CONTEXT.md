## Purpose
Cross-cutting admin, governance, and data-integrity surface. Covers identity
and tenancy (user / workspace / org administration, access control,
ownership transfer), platform-operational trust (backup, restore, audit
log), and the evidentiary spine that every analysis layer hangs off
(provenance notes, source citations, assumption tracking, confidence
ratings, incomplete-data warnings, manual-override logging, QA + design
review checklists, locked approval states, project archive / recovery,
naming conventions, metadata management). This is the section a reviewer
opens when they need to trust — or contest — an Atlas output.

## Known gotchas
- **Existing code overlap.** RBAC (`apps/api/src/plugins/rbac.ts`) already
  implements access controls — reuse, do not duplicate. Org + member
  routes (`apps/api/src/routes/organizations/`, `routes/members/`)
  already cover user / workspace / org settings at the P1 bar.
- **Audit log is split.** The persisted log lives in
  `apps/api/src/routes/activity/` + `apps/api/src/lib/activityLog.ts`;
  this section is a consumer, not the owner. Do not introduce a second
  log pipeline.
- **Confidence is a type, not a UI feature.** `WithConfidence<T>` is
  already threaded through the scoring engine — governance views render
  the existing field, they do not compute it.
- **Incomplete-data warnings are already a widget.** `DataCompletenessWidget`
  exists in the assessment feature folder. Compose it; do not reimplement.
- **Project ownership transfer is unimplemented.** Any transfer flow must
  route through RBAC + activity log to preserve the audit trail — it is
  intentionally `planned` until a migration lands.
- **Approval state is not a boolean.** The future-state model will have
  `draft | in-review | approved | locked | archived` — reserve the schema
  shape before wiring UI.
