## Purpose
Section 17 is the rules engine behind the Atlas design suggestions: a
declarative layer of siting, siting-conflict, protection, and
priority-weighting rules that powers the "why this suggestion was made"
explanations surfaced throughout Sections 8-16. It gives designers an
explainable, editable scoring model — structure / orchard / water /
grazing / quiet / sacred zone siting rules, guest privacy and safety
buffers, ecological and wildlife protections — plus preset project
templates (farm-only, retreat / education, conservation-first, homestead,
multi-enterprise) that seed the weights. Users adjust priority sliders
and the map re-scores candidate placements in place.

## Known gotchas
- **Legacy folder collision.** `apps/web/src/features/rules/` pre-exists
  from earlier scaffolding and holds legacy rule code. The canonical
  home for Section 17 going forward is this folder (`design-rules/`).
  Do not attempt to rename or collapse the legacy folder as part of
  this section's rollout — that is a separate migration task. When
  porting real logic, copy from `features/rules/` into `features/design-rules/`
  incrementally and delete the legacy source in a single follow-up PR.
- **MT gating.** The `ogden-specific-design-template` item is phase `MT`
  and must only render when `ATLAS_MOONTRANCE=1`. Use the feature-manifest
  helpers (`isFeatureVisible`) rather than ad-hoc env checks.
- **Explainability contract.** "Why this suggestion was made" output must
  comply with Section 0d AI guardrails — no freeform LLM narration without
  a rule-trace backing it. Every surfaced suggestion needs a deterministic
  chain of fired rules attached.
- **Weight sliders are per-project, not global.** Templates seed weights
  but user overrides must persist per-project, never leak across projects.
