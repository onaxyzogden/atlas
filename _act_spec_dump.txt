1. Context & architectural decisions



1.1 Act in the Plan–Act–Observe cycle

Act is the execution stage of OLOS. It converts approved Plan handoff packages into field work, records proof of completion, and feeds verified data into the Observe layer. Act is not for designing, redesigning, or making planning decisions — those belong in Plan.



Core principle

Observe records reality. Plan makes decisions. Act gets the right work done, proves it, and feeds learning back into the system.



field actions are generated by Plan objectives. Two categories of field action exist, but they are treated identically in the UI:



Category

Description

Example

Survey / observation tasks

Field data collection that feeds the Observe layer. Generated by Tier 0–2 Plan objectives.

Topographic survey walk, soil jar test, sector compass observation

Implementation tasks

Physical changes to the land. Generated by Tier 3–6 Plan objectives.

Dig swale, establish windbreak, install water point, plant guild



Why identical UX treatment

From the steward’s perspective in the field, both task types involve the same actions: go to a location, do the work, record proof, mark done. Distinguishing them in the UI adds cognitive load without adding value. The distinction matters for data routing (survey tasks feed Observe differently than implementation tasks) but that is handled in the data layer, not the interface.



1.2 Two views, one Command Center

The Act Command Center has two views. They are not separate pages — they are two modes of the same surface.



View

View B is the default because returning users — the majority of Act interactions — do not come from Plan. They come back after a day in the field, after a week away, or after a contractor has been on site. They need the full picture first.



View A is the drill-down. It narrows the surface to one objective’s tasks, proof requirements, and map context. It is always reachable from View B by selecting an objective, and it is the view launched by the “Launch Act Command Center” button in Plan.



1.3 Act is mobile-first

Plan is primarily a desktop/tablet surface used at a desk or in a planning session. Act is primarily a mobile surface used in the field — muddy hands, bright sunlight, unreliable connectivity. Every design decision for Act must be made with this context in mind.

3. Proof schema



3.1 Principles

Each field action carries a defined minimum proof schema. The schema specifies what evidence is required before the task can be submitted. Additional evidence beyond the minimum is always permitted.



A universal catch-all schema (any combination of photo/GPS/note) is insufficient. It produces incomplete evidence records that degrade the quality of data feeding into Observe. Each task type must define its minimum explicitly.



Proof quality principle

The Observe layer is only as good as the evidence that feeds it. Weak proof schemas produce a convincing-looking Observe dashboard built on incomplete data. Every proof schema must be designed with the Observe output in mind — what does this evidence actually prove, and is that enough to update the relevant Observe data point with confidence?

3.4 Proof UI behaviour

Required proof items are shown as unfilled slots at the top of the task detail view.

Each slot shows its type, a brief instruction, and a capture button (camera icon for photo, crosshair for GPS, ruler for measurement, form icon for logged result).

A slot turns green and shows a thumbnail or value when the requirement is met.

Additional evidence can be added below the required slots via an “Add more evidence” control.

The submit button is disabled until all required proof slots are filled.

On mobile, camera and GPS capture launch native device capabilities directly — no intermediate steps.



Offline behaviour

Core proof capture must work offline. The field is the primary Act environment and connectivity cannot be assumed. The following rules reconcile the proof UI with offline reality:

Proof slots fill locally on capture. The UI does not require a network round-trip to confirm a slot is filled.

The submit button activates based on local slot state, not server confirmation.

Submitted tasks are queued locally when offline. A sync indicator in the top bar communicates pending uploads.

GPS coordinates are captured from the device’s local GPS hardware — no network request required.

Photos are stored to local device storage pending sync. Thumbnails render from local storage.

On reconnection, all pending proof uploads and status transitions sync automatically. Conflict resolution is handled by the offline sync infrastructure spec (deferred — see section 10).

Tasks cannot be verified in review mode while offline. The submitted state persists locally and verification triggers on sync.



Implementation constraint

The proof UI behaviour described above assumes a local-first data model for Act. Any architecture that requires server confirmation before slot state updates will break the field experience. The offline sync infrastructure spec must be in place before Act proof capture is implemented.




4. View B — All Tasks Dashboard



4.1 Purpose

View B answers the question: “What do I have outstanding across the whole project right now?” It is the default entry point for Act and the surface returning users land on.



It must not overwhelm. A project at mid-execution may have 40+ tasks across 10 objectives. View B’s job is to surface what is actionable today, not to display everything at once.



4.2 Layout

View B uses a single-column layout on mobile. On tablet and desktop it may use a two-column card grid. No three-panel layout — Act does not have the same information density needs as Plan.

The filter control allows the user to narrow the dashboard by:

Filters persist within the session. A “Clear filters” control is always visible when filters are active.

Proof slots (primary content)

Proof slots are the primary content of the expanded task — equivalent to the checklist’s role in Plan. They must have the highest visual weight. See section 3.4 for proof UI behaviour.



Checklist steps (secondary)

Some tasks carry sub-steps that appear below proof slots. Each sub-step carries a ‘feeds into →’ consequence tag where applicable, matching the Plan spec’s feeds-into pattern. The tag shows the downstream Observe data point this step contributes to. Their weight differs by task type:



Task type

Checklist step treatment

Rationale

Survey / observation tasks

Steps are proof methodology. They define how the evidence must be collected to be valid. Steps should be completed before submitting — the UI strongly encourages this with a warning on early submission, but does not hard-gate.

For a topographic survey, ‘mark high/low points’ and ‘record slope gradients’ are not optional guidance — they define what the survey consists of. Incomplete steps mean incomplete evidence.

Implementation tasks

Steps are operational guidance. The task can be submitted once proof requirements are met regardless of checklist state.

For earthworks, a checklist step like ‘compact the berm edges’ is a reminder, not a proof requirement. The photo and GPS trace are what prove the work happened.



The distinction is surfaced to the developer via a boolean field on the ChecklistStep object: “is_methodology: true” marks steps that carry evidential weight. The UI renders these with a stronger visual treatment and shows a soft warning if the user attempts to submit with methodology steps incomplete.

5.4 Act map view



5.4.1 Purpose

The Act map view is the primary execution surface for most field tasks. A steward executing a terrain survey, walking a fence line, or documenting an earthwork is working spatially — the task list and proof slots are supporting context, not the primary interface. View A without the map is incomplete.



View A ships with the map view or not at all

A steward executing a survey task using only the task list and proof slots, without spatial context, is working half-blind. The map view is not an enhancement to View A — it is the environment in which Act work actually happens. Implementation of View A and the Act map view must be treated as a single unit.



5.4.2 Layout

The map fills the full screen. All task context floats over the map as a bottom sheet (mobile) or a right panel (tablet/desktop). There is no split-screen layout — the map is primary, everything else is overlay.

5.4.3 Overlay loading

Overlays load automatically based on the active objective’s overlay_ids[]. The steward does not select overlays manually in Act — the system loads what is needed for the work at hand.



Overlays are loaded in the same set defined by the parent Plan objective.

The steward can toggle individual overlays on or off via a layers control in the top bar — but the default state is all objective overlays active.

No overlays outside the objective’s defined set are accessible in Act map view. The steward is not in a browsing mode; they are in an execution mode.

On offline map load, overlays are rendered from cached tile data for the project area. Overlay data cached at last sync.

5.4.5 GPS walk mode

GPS walk mode is the primary spatial capture method for survey tasks that require tracing paths, boundaries, or areas. It is the Act equivalent of drawing a contour line — the steward walks the feature and the device records the trace.



Activated by tapping the GPS walk tool. A countdown (3 seconds) gives the steward time to orient before recording begins.

While walking, a live trace renders on the map in real time. Elapsed distance shown in the GPS walk indicator.

Pause is available for rest breaks, gates, or obstacles. The trace is suspended but not discarded.

Stop and save commits the trace to the proof slot. The steward is shown the completed trace and prompted to confirm or discard.

Minimum trace accuracy threshold: if GPS signal drops below acceptable accuracy during a walk, a warning appears and the affected segment is flagged. The steward can accept or retrace the flagged segment.

GPS walk works fully offline using device hardware. Trace data is stored locally and synced on reconnection.



5.4.6 Proof pins on the map

As proof items are captured, they appear as pins on the map at their GPS location. This gives the steward a spatial record of their evidence and makes gaps visible at a glance.

5.4.8 Transition between map view and task list

Entering the map: tapping the map strip in View A opens the full map view. The active task is pre-loaded.

Returning to task list: tapping ‘Back to task list’ in the top bar returns to View A with the same task selected and proof state preserved.

The map view and View A are the same session — proof items captured on the map appear immediately in View A’s proof slots, and vice versa.

There is no separate navigation step between map and task list. They are two views of the same active task.



5.4.9 Offline map tile caching

Map tiles for the project area are cached on first project load and refreshed on subsequent syncs.

Cache area is defined by the project’s site boundary polygon, expanded by a configurable buffer (default 500m).

Stewards can manually trigger a tile cache refresh from the project settings before heading into the field.

When offline, a ‘Offline map’ indicator appears in the top bar. Tiles outside the cached area show a ‘Not available offline’ placeholder.




6. Reality diverges path



6.1 Design principle

Divergence from plan is not an edge case in land stewardship. It is a routine event. A steward who discovers unexpected bedrock, an unmarked spring, contaminated soil, or a structural constraint that makes a designed intervention unworkable needs a first-class path to record what they found and push it back to Plan — without losing their work, breaking other tasks, or having to leave the Act surface.



This is where most land management tools fail

Tools that treat plan divergence as an error state — rather than a normal part of working with living systems — produce two bad outcomes: stewards who improvise in the field without updating the plan, and plans that slowly decouple from reality. The divergence path must feel as natural as submitting a completed task.



6.2 Divergence capture flow

When the steward taps “Reality diverges” on any task:



Task status moves immediately to diverged. The task is paused, not failed.

A divergence capture form opens as a bottom sheet (mobile) or side panel (tablet/desktop).

The steward documents what they found using the standard proof schema (photo + note minimum, GPS point recommended).

The steward names the divergence type from a short list (see 6.3).

On submission, a Plan revision flag is auto-generated and linked to the parent Plan objective and tier.

The Plan Command Center’s relevant tier and objective show a divergence indicator until the flag is resolved.

The divergence evidence is routed to Observe as a new observation — it is real ground truth regardless of what happens to the plan.



6.3 Divergence types

Divergence type

Description

Physical constraint

A feature of the land prevents the planned intervention (bedrock, existing structure, wrong soil type)

New discovery

Something not in the plan was found (spring, contamination, protected species, archaeological feature)

Access issue

The location cannot be reached as planned (track condition, locked gate, seasonal flooding)

Resource constraint

Materials, equipment, or labour required are unavailable or insufficient

Safety concern

Proceeding would create an unsafe condition for people, animals, or the land

Plan error

The Plan objective contains an error, omission, or assumption that does not hold in the field



6.4 Divergence state in the UI

A diverged task is visually distinct from blocked, in_progress, or submitted tasks. It must communicate urgency without alarm.



Status badge: amber “Diverged” with a refresh/loop icon (not red — this is not a failure)

Task card in View B shows the divergence type and the date flagged

The parent objective in View A shows a divergence indicator on its header

In the Plan Command Center, the relevant objective card gains a “1 divergence flag” indicator in amber

The diverged task remains in View B’s “Blocked & diverged” section until the Plan revision is approved and the task is re-opened



6.5 Resolving a divergence

Resolution happens in Plan, not in Act. The Plan revision flag generated by the divergence creates a new or modified objective in the relevant tier. When that objective is approved:

The diverged field action is either re-opened with an updated scope, replaced by new tasks, or formally retired

The divergence flag is resolved and removed from the Plan objective indicator

The divergence evidence remains in Observe permanently as a ground-truth record

7.2 Verification mode assignment

Verification mode is set at the task level by the Plan objective that generated the task. It can be overridden by the project steward in Act. It cannot be changed after a task has been submitted.



Contractor scope completion tasks always default to review mode

Tasks with logged_result proof items (soil tests, infiltration rates) are recommended for review mode where data quality is critical for Observe

Stewards can manually flag any self-mode task for review before submitting



7.3 Verifier UX

A verifier is a project member with verification rights. When a review-mode task is submitted:

The verifier receives a notification with the task title, proof items, and a direct link

The verifier can: Confirm (moves to verified, triggers Observe feed), Return for revision (moves back to in_progress with a note), or Escalate to Plan (equivalent to raising a divergence flag)

The submitted task is visible in View B under “Active tasks” with a “Pending verification” badge

Self-submitted tasks awaiting verification are visually distinct from in_progress tasks so the submitting steward can track their status




8. Observe feeds



8.1 How Act feeds Observe

The Observe layer is populated exclusively by verified field actions. A task in any state other than verified does not update Observe. This protects the integrity of the read-only intelligence layer.

Observe impact

not_started

None

in_progress

None

submitted (self mode)

Observe updated immediately

submitted (review mode)

None until verified

verified

Observe updated immediately

diverged

Divergence evidence routed to Observe as a new observation record (separate from the planned data point)

blocked

None



8.2 Divergence evidence in Observe

When a task is marked diverged, the captured evidence (photo, note, GPS) is routed to Observe as a new observation record under the relevant domain. This is intentional and unconditional — divergence evidence is real ground truth and belongs in Observe regardless of whether the Plan is revised.



The Observe layer must reflect reality, not the plan

If a steward discovers an unmarked spring and the task is marked diverged, that spring exists. It must appear in Observe: Hydrology & Water as an observation record even if the Plan has not yet been updated. Observe is a record of what is, not a reflection of what was intended.




9. Developer acceptance criteria



9.1 View B — Dashboard

Default entry point for Act. Shown when navigating to Act without a specific objective context.

Next up card reflects current project state — not static. Updates when task states change.

Tasks grouped correctly: active, ready to start, blocked/diverged (collapsed), completed today (collapsed).

Filter control persists selection within session. “Clear filters” visible when active.

Tapping a task card opens View A with the tapped task’s parent objective loaded and the task pre-selected.



9.2 View A — Objective Execution

Reachable from View B and from Plan’s “Launch Act Command Center” button.

“Back to all tasks” control always visible. Returns to View B without losing scroll position.

Map strip shows correct overlay count for the active objective. Tapping opens the map view.

Tasks listed in dependency order. Active task expanded by default.

Objective completion gate greyed until all tasks in the objective are verified.



9.3 Proof

Required proof slots shown as unfilled until captured. Submit button disabled until all required slots filled.

Camera capture on mobile launches native camera without intermediate screens.

GPS capture auto-resolves current location. Manual pin on map available as fallback.

Logged result forms show task-specific fields — not a generic note field.

Additional evidence can always be added beyond the minimum schema.



9.4 Task status transitions

not_started → in_progress: automatic on first proof item added or checklist step checked.

in_progress → submitted: steward taps “Submit task” with all required proof slots filled.

submitted → verified (self mode): immediate and automatic.

submitted → verified (review mode): requires explicit verifier confirmation.

Any status → diverged: steward taps “Reality diverges” and completes capture flow.

Any status → blocked: steward taps “Mark blocked” and names the blocking dependency.

verified tasks do not transition to any other state. They are permanent records.



9.5 Reality diverges

“Reality diverges” action available on every task regardless of current status.

Tapping opens divergence capture form immediately — no confirmation dialog.

Divergence type selection required before form can be submitted.

Photo + note required as minimum proof for divergence capture.

On submission: task moves to diverged, Plan revision flag auto-generated and linked to parent objective, divergence evidence routed to Observe.

Plan Command Center shows divergence indicator on the relevant objective card.



9.6 Observe feeds

Verified tasks (both self and review mode) trigger Observe data point updates immediately on verification.

Diverged tasks route evidence to Observe as observation records immediately on divergence capture submission.

No other task states trigger Observe updates.

OLOS Act Stage Developer Specification

OLOS Act Stage Developer Specification

Domain → Objective → Assignment Checklist → Map Overlays → Proof → Feedback Loop



Purpose of this document

This specification defines the Act stage of OLOS as a developer-ready workflow. It keeps Act focused on converting approved Plan handoff packages into assigned work, field execution, verification, maintenance rhythms, escalation, and feedback. Act should not redesign the project or collect unstructured observations unless those observations are tied to work completion, blocked work, or stewardship feedback.

Core principle:

Observe records reality. Plan makes decisions. Act gets the right work done, proves it, and feeds learning back into the system.

Prepared for developer implementation of the task-focused OLOS map view component.

1. Executive Summary

OLOS is organized around three main stages: Observe, Plan, and Act. Each stage uses the same land-based Domains, but the user’s relationship to each Domain changes by stage. Act is where approved plans become coordinated reality. It is not for inventing new designs, skipping unresolved constraints, or drowning the user in every property layer. Act is for assigning work, preparing the field context, executing approved scopes, recording proof, verifying completion, maintaining ongoing systems, and sending feedback back into Observe or Plan when reality changes.

Developer takeaway

When the user selects Act + Domain + Objective, the map view should load only the overlays, approved Plan package, assignment fields, execution checklist, proof requirements, and escalation controls required for that active work. The user should see the work they can complete now, not the entire project universe.

3. Act Stage Boundary Rules

Act definition

Act is the execution, verification, maintenance, and feedback stage of OLOS. Its purpose is to turn approved Plan handoff packages into assigned work and stewardship routines, then prove what was completed, what was blocked, what changed, and what needs to be observed or replanned.

Allowed Act verbs

Assign

Schedule

Prepare

Execute

Install

Build

Repair

Maintain

Inspect

Verify

Log

Report

Pause

Escalate

Reopen

Complete

Archive

Repeat

Avoid in Act

Creating new design decisions without an approved Plan update

Executing work with unresolved high-risk or compliance blockers

Treating missing proof as completed work

Changing scope without recording the change and routing it for review

Collecting random observations unrelated to the active task or stewardship routine

Letting tasks stay assigned without owner, due date, status, or blocker reason

Replacing professional inspection, engineering, legal, ecological, or safety review when required

Act outputs instead of vague completion

Every Act task should resolve into a verifiable status: Verified Complete, Completed Pending Verification, Blocked, Needs Rework, Escalated to Plan, Escalated to Observe, Ongoing Stewardship, or Rejected/Cancelled. “Done” is not enough unless the proof matches the success criteria.

4. Map View Behavior

The Act map view should be a field execution workspace. It should show the selected Domain, the specific Objective, the approved Plan handoff, the active work area, dependencies, access, hazards, assignment details, checklist items, and proof requirements. It should not behave like a full GIS dashboard unless the task requires it.

5. Universal Overlays

Overlay

Purpose in Act

Zones

Frequency of human attention and maintenance intensity by area. Helps Act place daily/weekly work near appropriate stewardship zones.

Sectors

Incoming forces such as sun, wind, storms, wildfire path, wildlife corridors, noise, road exposure, smoke, odour, views, and neighboring influence. Used when execution must respond to directional forces.

Contours / Landform

Slope, aspect, elevation, high points, low points, ridgelines, valleys, depressions, and erosion paths. Used for earthwork, access, drainage, safety, and placement tasks.

Water Flow

Runoff, water entry/exit points, wet areas, dry areas, drainage, ponds, wells, springs, roof catchment, irrigation, and flood-prone areas.

Soil Conditions

Soil type, compaction, infiltration, erosion, contamination, organic matter, test zones, and soil health indicators.

Ecology / Habitat

Vegetation communities, wildlife corridors, invasive species, sensitive habitat, pollinator areas, wetlands, and restoration areas.

Access / Movement

Roads, paths, gates, parking, emergency access, equipment routes, animal lanes, delivery points, and harvest routes.

Infrastructure / Utilities

Buildings, fences, greenhouses, barns, storage, utilities, water lines, power, compost areas, and other physical assets.

Resource Flows

Energy, water, compost, manure, mulch, feed, greywater, waste, materials, and storage flows.

Roles / Responsibility

Stewards, owners, workers, volunteers, decision rights, permissions, accountability zones, and reporting responsibilities.

Risk / Compliance

Zoning, setbacks, permits, easements, fire risk, flood risk, contamination, protected areas, safety hazards, and legal constraints.

Suitability

Suitability or constraint status for dwelling, growing, grazing, building, access, restoration, water storage, or retreat use.

Stewardship Intensity

Care frequency: daily, weekly, monthly, seasonal, annual, passive/self-regulating, or unknown.

Monitoring / Records

Observation points, photo points, logs, tests, sensor readings, receipts, inspections, proof records, and before/after documentation.

Timeline / Phasing

Existing condition, immediate constraints, current phase, next phase, dependencies, deadlines, full vision, and ongoing stewardship.

8. Act Domain Specifications

The following specifications define a default Act objective for each universal Domain. Additional objectives can be added later, but each should follow the same structure and boundary rules.

1. Vision & Project Intent

Keeps execution aligned with the approved project intent and prevents scope drift during implementation.

Field

Developer Specification

Act objective

Execute only the approved phase of the project vision and flag any work that no longer matches the intent.

Working question

Is the active work still serving the approved land-based vision, phase, and non-negotiables?

Required Plan inputs

Approved project intent; phase plan; success criteria; non-negotiables; rejected or deferred items; risk/compliance gates

Default map overlays

Timeline / Phasing; Roles / Responsibility; Monitoring / Records; Risk / Compliance; Suitability

Execution checklist

Load current approved phase and related objectives

Confirm active task belongs to approved scope

Display non-negotiables and disallowed actions

Flag requests that create scope drift

Route drift, contradiction, or vision conflict back to Plan

Record who approved any phase-level change request

Evidence / proof required

Task-to-vision alignment note; approval reference; scope-change request if needed; supporting screenshots or notes

Completion criteria

Work is either confirmed aligned with the approved phase or routed for replanning before execution continues.

Feedback / escalation output

Vision Drift Flag, Scope Change Request, or Confirmed Alignment record routed to Plan or project governance.

Allowed statuses

Ready for Assignment; In Progress; Blocked by Scope Drift; Escalated to Plan; Verified Aligned; Cancelled



2. Land Base & Legal Context

Executes approved site-reference, boundary, legal-access, and physical-container tasks without turning legal interpretation into field improvisation.

Field

Developer Specification

Act objective

Field-verify and maintain approved boundary, access, setback, easement, and restricted-area information.

Working question

Can the active work happen in the correct legal and spatial container without crossing a known constraint?

Required Plan inputs

Approved boundary/access plan; legal-context notes; setbacks; easements; restricted areas; required reviews; location geometry

Default map overlays

Risk / Compliance; Suitability; Access / Movement; Infrastructure / Utilities; Monitoring / Records

Execution checklist

Confirm task location against boundary and legal layers

Verify access point or work area in the field

Mark boundary markers, gates, signs, or restricted-area indicators as installed/checked

Upload field photos or documents

Pause work if location conflicts with setback, easement, or protected-area layer

Escalate unresolved legal ambiguity before work proceeds

Evidence / proof required

Geotagged photos; marked boundary/access confirmations; document attachment; issue notes; verification sign-off where required

Completion criteria

Legal/spatial reference task is verified or blocked with a clear reason and required next action.

Feedback / escalation output

Boundary/Access Verification Record or Constraint Escalation routed to Risk/Compliance or Plan.

Allowed statuses

Assigned; In Progress; Blocked; Completed Pending Verification; Verified Complete; Escalated

9. Developer Acceptance Criteria

Product rule

When a user selects Act + Domain + Objective, OLOS should load only the overlay bundle required to complete that execution or stewardship task. The Domain determines the default overlay bundle. The Objective can add or remove overlays from that bundle. The approved Plan package determines what is allowed to be assigned.

Functional acceptance criteria

Act tasks must be created from Approved or Conditionally Approved Plan handoff packages unless explicitly marked as emergency/risk response.

Each Act objective must declare required Plan inputs, assignment fields, proof requirements, and verification rules.

If required Plan inputs are missing, the objective should show Not Ready for Act rather than allowing vague assignment.

Each task must have an owner or responsible role, status, location, checklist, and blocker field.

Each checklist item should link to at least one relevant overlay, field action, proof requirement, or verification criterion.

Completion cannot be final until required proof is attached and verification rules are satisfied.

Blocked work must require a blocker reason and escalation destination.

Scope changes discovered during Act must route back to Plan instead of silently changing the approved work package.

Unexpected conditions should route to Observe or Monitoring, not disappear inside task notes.

Compliance, safety, legal, animal welfare, ecological, and infrastructure-sensitive tasks must support pause and professional-review gates.

UX acceptance criteria

The map should open with the focused overlay bundle needed for the active task, not every project layer.

The side panel should show Stage, Domain, Objective, assigned steward, due date, status, approved scope, checklist, proof requirements, and blockers.

Users should be able to submit proof directly from the map task view.

Users should be able to mark work as blocked without pretending it is complete.

Completed Pending Verification and Verified Complete should be separate states.

Ongoing stewardship routines should be repeatable and schedulable, not one-off tasks only.

New field reality should be easy to route back to Observe, Plan, Risk, or Monitoring.

A person entering the project later should be able to see what was assigned, what was done, who verified it, and what changed because of it.

Clean boundary statement

Observe answers: What is happening? Plan answers: What should happen, where, why, and in what sequence? Act answers: Who does it, when, with what proof, what is blocked, and what feedback must return to the system?

1. System overview



1.1 What OLOS is

OLOS is the full operating system for regenerative land development. Its success is measured by whether it can independently guide a project through observation, design, phased implementation, daily management, ecological monitoring, and adaptive stewardship at the complexity level of Apricot Lane Farms — without requiring external project management systems or personnel to hold the work together.



1.2 The three-stage cycle

All work in OLOS moves through a recurring spiral of three stages:



Stage

Role in the system

Plan

Determines what needs to be known and done. Sequences objectives by dependency. The entry point for every new project and every new cycle. Generates Act tasks.

Act

Executes all field work — survey tasks and implementation tasks are treated identically in the UI. Collects proof. Feeds verified data into Observe. The primary mobile surface.

Observe

Read-only intelligence layer. Surfaces synthesised data produced by Act. Grows richer across cycles. Feeds back into Plan when conditions change enough to warrant revision.



The cycle is a spiral, not a sequence

Plan → Act → Observe repeats. Each cycle produces ground truth that informs the next Plan revision. The system is designed for this from the data model up — every data point carries a cycle ID, every Plan objective has a review mode for subsequent cycles, and the Observe Plan revision banner closes the loop automatically.

3. Ratified architectural decisions



The following decisions were made during spec development and are binding for implementation. They resolve questions that would otherwise be re-litigated during build.



Navigation & information architecture

Tier unlock transitions display a full-width celebration banner in the centre panel with a ‘What just became possible’ section. The lock icon animates away on unlock. This is the primary moment of felt progress.

Locked tiers open a popover (not a modal) with lock reason, prerequisite progress indicator, and a subtle ‘open anyway’ bypass link. An amber out-of-sequence banner appears if the user proceeds.

Domain is a tag on an objective, not a navigation gate. The user never selects a domain. Overlays load automatically based on what the objective requires.

Stage navigation is Stage → Objective, not Stage → Domain → Objective.

The 7-stage lifecycle is deprecated. The 3-stage cycle (Plan / Act / Observe) is the forward IA.

Tier 3 is named ‘Foundation Decisions’, not ‘First Design Decisions’. The name is cycle-neutral by design.



Plan stage

The detail panel is organised into four explicit sections: OBJECTIVE, MAP ACTIVATION, YOUR DECISIONS (with feeds-into tags), and REFERENCE (collapsed by default).

Each decision item carries a ‘feeds into →’ tag showing its downstream consequence. Tags are authored per item via the feeds_into field on ChecklistItem. When checked, the tag briefly highlights green.

Cyclical review entry shows a modal explaining what Observe detected and which objectives are flagged. Confirmation shows a ‘Decision confirmed’ or ‘Decision updated’ badge and a summary line.

Tier gate logic has two modes: initial completion (hard gates) and cyclical review (soft gates — previously completed tiers become review checkpoints).

Objective status is computed, not manually set. Resolution order: complete → active → available → locked.

Plan outputs are one of: Approved for Act, Conditionally Approved, Needs More Observation, Needs Professional Review, Redesign Required, Deferred, Rejected.



Act stage

The Launch Act Command Center transition collapses the left sidebar to objective-relevant tools only, pre-activates overlays on the map, and shows a floating bottom-right card with objective name, decision progress, and a Back to Plan button.

Survey tasks and implementation tasks are treated identically in the UI. The distinction matters for data routing only.

Proof schemas are task-type specific. A universal catch-all schema is explicitly rejected.

Checklist steps in survey tasks are proof methodology (is_methodology: true) and carry evidential weight. In implementation tasks they are operational guidance.

Act is mobile-first. The three-panel Plan layout must not be applied to Act.

View A (objective execution) must ship with the Act map view. They are a single unit.

Objective overlays are the default active set on map open. The full project overlay library remains accessible via the layers control.

Reality diverges is a first-class action available on every task. It is not an error state.

Divergence evidence routes to Observe unconditionally and permanently, regardless of Plan revision status.



Observe stage

Observe is read-only. It is populated exclusively by verified Act tasks and divergence captures.

No health scores, traffic-light ratings, or aggregated wellness indices. Observe shows evidence; it does not evaluate it.

Observe data is never edited or deleted from within the Observe surface. Corrections require a new verified Act task that supersedes the prior data point.

Superseded data points remain visible and accessible. They are never hidden.

The Plan revision banner triggers are tiered: Critical (undismissable), High (dismissible with reminder), Informational (summary row only).

Presentation mode excludes all internal planning state. It surfaces verified evidence only.



Project Home

The Next Up card surfaces exactly one item. Showing multiple items hands the prioritisation decision back to the steward, defeating the card’s purpose.

Urgency score is never displayed as a number.

Contractor access is enforced at the data layer. It is not a UI-only restriction.

Contractor access expires under three conditions: all tasks verified, manual revocation, or 90-day time fallback.



Notifications

Interrupt only when the cost of delay is concrete.

Critical-tier notifications cannot be opted out of. Only delivery address can be changed.

Informational notifications never trigger email or push.

Safety divergence and protected species discovery notifications are delivered to all project team members regardless of role — the safety override crosses all role boundaries.

Escalation updates the existing notification in-place rather than creating a new entry.

The 14-day inactivity notification defaults to email on for primary stewards. It is the one Informational event with email default-on.

5. Recommended build sequence



The following sequence minimises blocked work and surfaces value to real users at each milestone.

Build targets

Phase 2

Multi-project management. The spiral closes — Observe feeds back into Plan across cycles.



Phase 1 is the make-or-break phase

If a steward can create a project by walking their boundary, name their land, and see their first four survey objectives ready to start — all in under 10 minutes — the system has earned trust. Every subsequent phase builds on that trust. If Phase 1 feels slow, bureaucratic, or spatially disconnected, later phases will not recover it.




6. Open questions for implementation team



The following questions were intentionally left to the implementation team. They are infrastructure or product decisions that fall outside UX spec scope.



Question

Context

What is the offline sync conflict resolution strategy?

Act spec requires local-first data model. When two team members modify the same task offline simultaneously, which write wins? Last-write-wins is simplest but may discard proof items.

What map tile provider is used?

Act map view and wizard boundary drawing depend on a tile provider. Provider affects offline caching strategy, usage costs, and available satellite imagery quality.

What is the GPS accuracy threshold for walk-the-boundary mode?

Act spec notes an accuracy threshold for GPS walk traces but defers the specific value. Recommend ±10m for boundaries, ±5m for survey traces.

How are cycle IDs generated and scoped?

Observe spec defines cycle IDs as immutable once assigned. The triggering condition for a new cycle (steward initiates a new Plan stage after Act completion?) needs engineering definition.

What is the maximum project site area supported?

No upper bound is defined. Very large stations (50,000+ ha) may require tile caching strategies different from small holdings.

How are measurement units handled globally?

Act proof schema uses measurements with units. The system should respect locale defaults (metric vs imperial) but allow per-project override.
